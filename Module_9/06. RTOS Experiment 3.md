## Experiment 3: Coordinated Motor Control with Semaphores

This experiment brings everything together. Three sensor/actuator tasks run concurrently -- the IMU drives the servo, the VL53L1X supervises the stepper, and the stepper runs a continuous trapezoidal motion profile until told to stop. Two binary semaphores coordinate the supervisor and the stepper: one signals a stop when an obstacle is detected, another signals a resume when the path is clear.

By the end, you'll have five concurrent tasks running on a single core, coordinated entirely through RTOS primitives -- no shared globals, no polling flags, no super-loop in sight.

---
### What You'll Need

- Integration Board with MPU6050, VL53L1X, PCA9685, and A4988 all seated
- Logic analyser connected to PE4 (STEP signal)
- All driver and utility files from Experiments 1 and 2 carried over

---
### How the Semaphores Work

A binary semaphore has two states: available (1) and unavailable (0). A task that calls `osSemaphoreAcquire()` on an unavailable semaphore blocks until another task gives it. This makes binary semaphores ideal for signalling between tasks -- one task waits, another fires the signal.

In this experiment, two semaphores coordinate the supervisor and stepper:

- `stopSemHandle` -- given by the supervisor when distance drops below 200mm. The stepper task acquires this between steps to check whether it should halt.
- `clearSemHandle` -- given by the supervisor when the obstacle is removed. The stepper task waits on this after halting, blocking until the path is clear.

The stepper never polls a flag or checks a variable. It simply blocks on a semaphore and resumes when the semaphore is given. This is the correct RTOS pattern -- blocking is free in terms of CPU cycles, whereas polling wastes them.

Declare the two semaphore handles below the existing mutex and queue declarations:

```c
/* Obstacle semaphores */
osSemaphoreId_t stopSemHandle;
osSemaphoreId_t clearSemHandle;
```

Add the stepper and supervisor task attributes alongside the existing thread attribute structures:

```c
const osThreadAttr_t stepperTask_attributes = {
    .name       = "StepperTask",
    .stack_size = 512 * 4,
    .priority   = (osPriority_t) osPriorityNormal,
};

const osThreadAttr_t supervisorTask_attributes = {
    .name       = "SupervisorTask",
    .stack_size = 512 * 4,
    .priority   = (osPriority_t) osPriorityAboveNormal,
};
```

The supervisor runs at `osPriorityAboveNormal` so it can preempt the stepper task promptly when an obstacle is detected. A supervisor that can be delayed by the task it's supervising defeats the purpose.

Add the two new task prototypes alongside the existing ones:

```c
void StepperTask(void *argument);
void SupervisorTask(void *argument);
```

Update `MX_FREERTOS_Init()` to create the semaphores and the two new tasks. Both semaphores are initialised with a count of zero -- neither is available until explicitly given:

```c
void MX_FREERTOS_Init(void) {
    rtos_utils_init();

    i2cMutexHandle = osMutexNew(&i2cMutex_attributes);

    imuQueueHandle = osMessageQueueNew(1, sizeof(MPU6050_Data), NULL);

    stopSemHandle  = osSemaphoreNew(1, 0, NULL);
    clearSemHandle = osSemaphoreNew(1, 0, NULL);

    osThreadNew(SupervisorTask, NULL, &supervisorTask_attributes);

```

### Writing the Task Functions

Add both task functions below the existing task functions in `freertos.c`.

**StepperTask** runs the motor in a continuous loop. Each iteration executes one full trapezoidal move. Between every step, it checks `stopSemHandle` with a zero timeout -- a non-blocking check that doesn't delay the step if no stop signal is pending. If a stop signal arrives, it halts immediately and waits on `clearSemHandle` until the supervisor clears it:

```c
void StepperTask(void *argument) {
    // Trapezoidal profile parameters
    // All delays in milliseconds -- 1ms resolution at 1000Hz tick rate
    const uint32_t TOTAL_STEPS  = 200;
    const uint32_t ACCEL_STEPS  = 50;
    const uint32_t MAX_DELAY_MS = 10;    // Slow start/end
    const uint32_t MIN_DELAY_MS = 1;     // Cruise speed

    for(;;) {
        rtos_print("[Stepper] Starting move -- %lu steps\r\n", TOTAL_STEPS);

        for (uint32_t step = 0; step < TOTAL_STEPS; step++) {

            // Non-blocking check for stop signal
            if (osSemaphoreAcquire(stopSemHandle, 0) == osOK) {
                rtos_print("[Stepper] Obstacle detected -- halting.\r\n");
                A4988_Disable(&stepper);

                // Block until supervisor clears the path
                osSemaphoreAcquire(clearSemHandle, osWaitForever);

                rtos_print("[Stepper] Path clear -- resuming.\r\n");
                A4988_Enable(&stepper);

                // Restart the move from the beginning
                break;
            }

            // Calculate delay for trapezoidal profile
            uint32_t delay_ms;
            uint32_t decel_start = TOTAL_STEPS - ACCEL_STEPS;

            if (step < ACCEL_STEPS) {
                // Acceleration -- delay decreases linearly
                delay_ms = MAX_DELAY_MS
                           - ((MAX_DELAY_MS - MIN_DELAY_MS)
                           * step / ACCEL_STEPS);
            } else if (step >= decel_start) {
                // Deceleration -- delay increases linearly
                uint32_t steps_in_decel = step - decel_start;
                delay_ms = MIN_DELAY_MS
                           + ((MAX_DELAY_MS - MIN_DELAY_MS)
                           * steps_in_decel / ACCEL_STEPS);
            } else {
                // Cruise
                delay_ms = MIN_DELAY_MS;
            }

            A4988_Step(&stepper);
            osDelay(delay_ms);
        }

        // Brief pause between moves before restarting
        osDelay(500);
    }
}
```

**SupervisorTask** polls the VL53L1X independently of `ToFTask`. It maintains a simple state machine -- path clear or obstacle present -- and gives the appropriate semaphore when the state changes:

```c
void SupervisorTask(void *argument) {
    const uint16_t OBSTACLE_THRESHOLD_MM = 200;

    uint8_t obstacle_present = 0;

    for(;;) {
        uint8_t ready = 0;

        osMutexAcquire(i2cMutexHandle, osWaitForever);
        VL53L1X_CheckForDataReady(VL53L1X_DEV, &ready);
        osMutexRelease(i2cMutexHandle);

        if (!ready) {
            osDelay(5);
            continue;
        }

        VL53L1X_Result_t result;
        VL53L1X_ERROR err;

        osMutexAcquire(i2cMutexHandle, osWaitForever);
        err = VL53L1X_GetResult(VL53L1X_DEV, &result);
        VL53L1X_ClearInterrupt(VL53L1X_DEV);
        osMutexRelease(i2cMutexHandle);

        if (err != 0 || result.Status != 0) {
            osDelay(5);
            continue;
        }

        if (result.Distance < OBSTACLE_THRESHOLD_MM && !obstacle_present) {
            // Obstacle just appeared
            obstacle_present = 1;
            rtos_print("[Supervisor] Obstacle at %u mm -- signalling stop.\r\n",
                       result.Distance);
            osSemaphoreRelease(stopSemHandle);

        } else if (result.Distance >= OBSTACLE_THRESHOLD_MM && obstacle_present) {
            // Obstacle just cleared
            obstacle_present = 0;
            rtos_print("[Supervisor] Path clear -- signalling resume.\r\n");
            osSemaphoreRelease(clearSemHandle);
        }

        osDelay(INTER_MEAS);
    }
}
```

The state variable `obstacle_present` ensures the semaphores are given exactly once per transition -- not repeatedly on every loop iteration while the obstacle remains. Without this, the semaphore count would accumulate and the stepper would receive multiple spurious signals.

---
### A Note on Shared VL53L1X Access

Both `ToFTask` and `SupervisorTask` poll the VL53L1X independently, and both call `VL53L1X_CheckForDataReady()` followed by `VL53L1X_GetResult()`. Since both calls are wrapped in the I2C mutex, the bus is protected. However, the two tasks will occasionally race to consume the same measurement -- whichever acquires the mutex first will read the result, and the other will find no data ready.

This is acceptable for this experiment. `ToFTask` exists to print readings for observation; `SupervisorTask` exists to make decisions. If the supervisor occasionally misses a print-side reading or vice versa, the system still behaves correctly. In a production design you'd consolidate all VL53L1X access into a single task and distribute results via a queue, the same way the IMU feeds the servo task.

### A Note on PE4 and Bit-Banging

You may have noticed that the stepper driver in this experiment works differently from what was originally planned. The original A4988 driver used a hardware timer to generate the STEP signal automatically -- a timer peripheral toggles the pin at a precise frequency, with no CPU involvement after the move starts. This is the correct approach for production stepper control, and it's what the driver was written for.

The problem is a PCB routing constraint on the Integration Board. PE4 -- where the STEP signal is wired -- is not connected to any timer output on the STM32F407. This was an oversight during layout, and without a board respin there's no way to fix it in hardware. The workaround is **bit-banging**: driving the pin entirely in software, with the CPU manually toggling it high and low, and `osDelay()` controlling the time between pulses.

Bit-banging is a general technique for implementing a hardware protocol in software when the dedicated hardware peripheral isn't available or isn't suitable. It's flexible -- you can bit-bang almost any signal on almost any GPIO -- but it comes at a cost. Timing precision is limited by the scheduler's tick resolution, which at 1000Hz gives a minimum delay of 1ms and a maximum step rate of 1000 steps per second. A hardware timer can achieve step rates orders of magnitude higher without consuming any CPU time. For a demonstration observed on a logic analyser, bit-banging is perfectly adequate. For driving a real motor at useful speeds, hardware PWM is the right tool.

This is an honest limitation of the board as built, and it's documented here rather than papered over. If you replicate this design, route your STEP pin to a timer-capable GPIO from the start.

On the logic analyser, PE4 will show the STEP signal as a burst of pulses with visibly varying frequency -- slow at the start, fast through the middle, slow again at the end. When the supervisor signals a stop, the pulses cease. When it signals a resume, they restart from the beginning of the trapezoidal profile.
