## "volatile" and the Motor Command Register

*"Volatile is not optional for hardware-facing variables."*

### The Problem with Trusting the Compiler

When you write embedded C, you are not the only one deciding how your code runs. The compiler is also making decisions, and it is very good at rewriting your code in ways that are faster or smaller, while producing the same observable result. The key word is observable. The compiler's definition of observable and yours are not always the same thing.

On a desktop machine this is rarely a problem. But in embedded systems, where variables can be changed by hardware, by interrupts, or by DMA transfers happening completely outside the normal flow of execution, the compiler's optimisations can silently break your program in ways that are extremely difficult to diagnose.

The volatile keyword exists to close this gap. It is a contract between you and the compiler: do not optimise accesses to this variable. Read it from memory every time. Write it to memory every time. Assume its value can change at any moment for reasons you cannot see.

Understanding when and why to use volatile is one of the most important habits in embedded C. This experiment makes the consequences of omitting it visible on real hardware.

### What Does the Compiler Actually Do Without volatile?

Consider this loop:

```c
uint32_t motor_speed = 0;

while (1) {
    if (motor_speed == 100) {
        printf("Speed reached\r\n");
    }
}
```

If motor_speed is only ever written in an ISR and read in the main loop, the compiler — at optimisation level -O2 — may look at this code and reason as follows:

"motor_speed is never written inside this loop. Its value cannot change here. I will read it once before the loop, store it in a CPU register, and use that register for every comparison. This is faster than reading from SRAM on every iteration."

The compiled result behaves as if you had written:

```c
uint32_t cached = motor_speed;   // Read once
while (1) {
    if (cached == 100) {          // Register comparison — never re-reads SRAM
        printf("Speed reached\r\n");
    }
}
```

The ISR updates motor_speed in SRAM. But the main loop is comparing against a stale register value and never sees the update. The condition never triggers. The motor command is silently lost.

Declaring the variable volatile prevents this entirely:

```c
volatile uint32_t motor_speed = 0;
```

Now the compiler is forbidden from caching the value. Every read goes to SRAM. Every write goes to SRAM. The ISR and the main loop are guaranteed to see each other's updates.

### Why is this relevant to robotics?

Motor control firmware is built around exactly this pattern. A timer ISR fires at a fixed rate -- typically every 1ms or faster,  and updates a shared variable representing the current speed, position, or PWM duty cycle. The main loop or a PID controller reads that variable and makes decisions based on it.

If the shared variable is not declared volatile, the compiler may cache it and the control loop operates on a frozen snapshot. The motor does not respond to commands. The PID loop does not converge. The robot behaves erratically or not at all, and the bug leaves no obvious trace because the code looks completely correct when you read it.

This class of bug is particularly dangerous because:

- It is optimisation-dependent. The code works fine at -O0 (debug builds) and silently breaks at -O2 (release builds).

- It is timing-dependent. Whether the bug manifests depends on how often the ISR fires relative to the main loop.

- It is invisible in the debugger. Attaching a debugger often forces memory reads, masking the cached register behaviour.

In short, it is exactly the kind of bug that passes all your tests and then fails in the field.

### What This Experiment Demonstrates

This experiment constructs a minimal motor control scenario. A timer ISR incrementing a shared speed variable, read by the main loop, and shows you the concrete difference volatile makes at the assembly level. By the end you will be able to:

- Observe the compiler silently caching a non-volatile shared variable.
- Confirm the bug by reading the disassembly inside STM32CubeIDE.
- Fix it with volatile and verify the fix in the same disassembly view.
- Understand exactly when volatile is required and why it cannot be omitted.

### What You Will Need

- STM32F4 Discovery board 
- STM32CubeIDE installed 
- USB Mini-B cable (for ST-LINK)
- A serial terminal (STM32CubeIDE's built-in console, or PuTTY/CoolTerm via a USB-UART adapter on PA2)

### Part 1 — Project Setup

1. Open STM32CubeIDE, go to File, New, STM32 Project.
2. Search for STM32F407VG, select the Discovery board, name the project (e.g. Exp2_Volatile), and click Finish.
3. In the .ioc configurator: Under Connectivity, enable USART2. Set mode to Asynchronous, baud rate 115200, 8N1. Under Timers, enable TIM2. Set Clock Source to Internal Clock. Under NVIC Settings, enable TIM2 global interrupt. Under System Core, SYS, set Debug to Serial Wire. Click Save and generate code.
4. Set the compiler optimisation level to -O2 so the caching behaviour is active. In STM32CubeIDE, right-click your project and open Properties. Navigate to C/C++ Build, Settings, MCU GCC Compiler, Optimization. Set the level to Optimize more (-O2). Click Apply and Close.

This step is critical. At -O0 the compiler does not cache variables and the bug will not appear. The entire point of this experiment is to observe what happens under real-world optimisation settings.

### Part 2 — Retarget printf to UART

Add the following to Core/Src/main.c just below the includes:

```c
/* USER CODE BEGIN Includes */
#include <stdio.h>
/* USER CODE END Includes */
```

Add the _write syscall redirect:

```c
/* USER CODE BEGIN 0 */
int _write(int file, char *ptr, int len) {
    HAL_UART_Transmit(&huart2, (uint8_t*)ptr, len, HAL_MAX_DELAY);
    return len;
}
/* USER CODE END 0 */
```

### Part 3 — Configure TIM2 in Code

STM32CubeIDE generates a MX_TIM2_Init() function but does not start the timer automatically. Add the timer start call inside the user code section in main(), after all the MX_ init calls:

```c
/* USER CODE BEGIN 2 */
HAL_TIM_Base_Start_IT(&htim2);   // Start TIM2 with interrupts enabled
/* USER CODE END 2 */
```
Then set the TIM2 period in the .ioc file so the ISR fires approximately every 1ms. Open the .ioc file and navigate to Timers, TIM2, Parameter Settings. Set Prescaler to 83, which gives a 1MHz timer clock from the 84MHz system clock. Set Counter Period to 999, which produces a 1kHz interrupt rate. Save and regenerate code.

### Part 4 — The Experiment Code

#### Shared variable declarations

Place these at the top of main.c in the private variables section:

```c
/* USER CODE BEGIN PV */

// Non-volatile version — subject to compiler caching
uint32_t motor_speed = 0;

// Volatile version — guaranteed fresh read every time
volatile uint32_t motor_speed_v = 0;

#define TARGET_SPEED  100
#define MAX_SPEED     200

/* USER CODE END PV */
```

#### The ISR

This fires every 1ms courtesy of TIM2. Place it in the user code section at the bottom of main.c:

```c
/* USER CODE BEGIN 4 */

void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim) {
    if (htim->Instance == TIM2) {
        if (motor_speed < MAX_SPEED)   motor_speed++;
        if (motor_speed_v < MAX_SPEED) motor_speed_v++;
    }
}

/* USER CODE END 4 */
```

#### The main control loop

Replace the contents of the while(1) loop:

```c
/* USER CODE BEGIN WHILE */

printf("=== Experiment 2: volatile Demo ===\r\n");
printf("Waiting for motor_speed and motor_speed_v to reach %d...\r\n\n",
       TARGET_SPEED);

uint8_t non_volatile_triggered = 0;
uint8_t volatile_triggered     = 0;

while (1) {

    if (!non_volatile_triggered && motor_speed == TARGET_SPEED) {
        printf("[non-volatile] motor_speed reached %lu\r\n", motor_speed);
        non_volatile_triggered = 1;
    }

    if (!volatile_triggered && motor_speed_v == TARGET_SPEED) {
        printf("[volatile]     motor_speed_v reached %lu\r\n", motor_speed_v);
        volatile_triggered = 1;
    }

    if (non_volatile_triggered && volatile_triggered) {
        printf("\r\nBoth triggered. Experiment complete.\r\n");
        break;
    }

    if (motor_speed_v > TARGET_SPEED + 20 && !non_volatile_triggered) {
        printf("[non-volatile] NEVER TRIGGERED — compiler cached the value!\r\n");
        printf("[non-volatile] motor_speed in SRAM is actually: %lu\r\n",
               motor_speed);
        break;
    }
}

/* USER CODE END WHILE */
```

### Part 5 — Expected Serial Output

At -O2 with optimisation enabled and the bug active:

```
=== Experiment 2: volatile Demo ===
Waiting for motor_speed and motor_speed_v to reach 100...

[volatile]     motor_speed_v reached 100
[non-volatile] NEVER TRIGGERED — compiler cached the value!
[non-volatile] motor_speed in SRAM is actually: 121
```

The volatile variable triggers correctly. The non-volatile variable never triggers in the main loop even though the ISR has been writing to it the entire time and its value in SRAM is well past the target. The compiler cached it in a register before the loop and never went back to SRAM.

At -O0 with no optimisation and the bug hidden:

```
=== Experiment 2: volatile Demo ===
Waiting for motor_speed and motor_speed_v to reach 100...

[non-volatile] motor_speed reached 100
[volatile]     motor_speed_v reached 100

Both triggered. Experiment complete.
```

Both trigger correctly because the compiler does not cache anything at -O0. This is exactly why bugs like this survive into release builds; they are invisible during debug-mode development and only emerge when optimisation is turned on.

Switch between -O0 and -O2 in your project settings and reflash to observe both outcomes on the same hardware.

### Part 6 — Reading the Disassembly

This is where the experiment moves from observation to understanding. The disassembly shows you exactly what the compiler generated and why the bug occurs at the machine instruction level.

1. Build the project at -O2 and start a debug session via Run, Debug, or F11.
2. Once paused at main(), open the disassembly view via Window, Show View, Disassembly.
3. Step into the while(1) loop and look for the comparison against motor_speed. At -O2 you are looking for a pattern like this:

```
; Non-volatile version at -O2
; The compiler loads motor_speed into r3 ONCE before the loop
LDR    r3, [pc, #offset]    ; Load address of motor_speed
LDR    r3, [r3]             ; Load value — happens ONCE outside the loop

loop_top:
    CMP    r3, #100         ; Compare cached register — never re-reads SRAM
    BEQ    triggered
    B      loop_top
```

```
; Volatile version at -O2
; The compiler re-reads motor_speed_v from SRAM on every iteration
loop_top:
    LDR    r3, [pc, #offset]    ; Load address of motor_speed_v
    LDR    r3, [r3]             ; Load value from SRAM — every iteration
    CMP    r3, #100
    BEQ    triggered
    B      loop_top
```

The difference is a single extra LDR instruction per loop iteration, but that instruction is the difference between a working motor controller and one that never responds to speed commands. The exact register names and offsets will differ on your build, but the pattern is the same: the non-volatile version has its SRAM load hoisted out of the loop. The volatile version keeps it inside.

### Part 7 — The Fix

The fix is a single keyword. Everywhere a variable is shared between an ISR and the main loop, or mapped to a hardware register, declare it volatile:

```c
// Before — invisible bug waiting for -O2
uint32_t motor_speed = 0;

// After — compiler forbidden from caching
volatile uint32_t motor_speed = 0;
```

Rebuild at -O2 and reflash. Both variables now trigger correctly at the target speed, and the disassembly confirms the SRAM load has moved back inside the loop.

### Part 8 — When Does volatile Actually Apply?

A variable must be declared volatile if any of the following are true:

- It is written by an ISR and read by the main loop. For example a motor speed, encoder count, or sensor ready flag.
- It is mapped to a memory-mapped hardware register, for example a GPIO output register, timer capture-compare register, or UART data register.
- It is written by DMA and read by the CPU, for example an ADC sample buffer or UART receive buffer.
- It is shared between RTOS tasks without a mutex, for example shared sensor state in a FreeRTOS application.
- It is used as a busy-wait polling flag, any pattern of the form while (!data_ready) that waits for a flag set elsewhere.

### What You Should Take Away

The non-volatile variable never triggers at -O2 because the compiler cached it in a register and never re-read SRAM. The volatile variable triggers correctly because volatile forces a fresh SRAM read on every access. Both work correctly at -O0, which is why the bug is invisible in debug builds and only surfaces under optimisation. The disassembly shows one extra LDR per iteration for the volatile version, a real but tiny runtime cost that is always worth paying. The ISR was updating SRAM correctly the entire time. The bug was never in the ISR. It was in how the main loop read the variable.

The rule is simple: any variable that can be modified outside the normal flow of execution must be declared volatile. On an embedded target, outside the normal flow means hardware, interrupts, and DMA, all of which are present in virtually every robotics system you will ever write. Treating volatile as optional is not a style choice. It is a bug waiting for your release build.
