## Fixed-Width Types In a Sensor Struct

*"Size your types deliberately."*

### The Hidden Cost of Lazy Typing

When you write int or long in an embedded C program, you are not specifying a size. You are asking the compiler to pick a size that is convenient for the target architecture. On a 32-bit ARM Cortex-M4 like the STM32F4, int is 32 bits. On an 8-bit AVR it is 16 bits. On a 64-bit desktop it is still 32 bits but long becomes 64 bits. The size of these types is platform-defined, not language-defined, and it changes when you move your code between targets.

In isolation this sounds like a minor portability concern. In practice it has direct consequences for memory consumption, struct layout, wire protocol correctness, and the behaviour of arithmetic on sensor data. On an embedded target with kilobytes of RAM and a fixed communication protocol, these consequences are not theoretical. They show up as wasted memory, misaligned structs, corrupted sensor readings, and communication failures between devices.

The fix is straightforward. The C99 standard introduced the fixed-width integer types in stdint.h, and using them is one of the simplest habits that separates careful embedded code from careless embedded code.

### What Are Fixed-Width Types?

The fixed-width types are defined in stdint.h and their sizes are guaranteed regardless of platform or compiler:

- uint8_t is an unsigned integer exactly 8 bits wide. Range 0 to 255.
- int8_t is a signed integer exactly 8 bits wide. Range -128 to 127.
- uint16_t is an unsigned integer exactly 16 bits wide. Range 0 to 65535.
- int16_t is a signed integer exactly 16 bits wide. Range -32768 to 32767.
- uint32_t is an unsigned integer exactly 32 bits wide. Range 0 to 4294967295. 
- int32_t is a signed integer exactly 32 bits wide. Range -2147483648 to 2147483647.

Each of these has a precise, documented range. When you choose one for a variable you are making an explicit statement about what values that variable can hold. When someone else reads your code they know immediately what the type can contain without needing to know the target architecture. When you move the code to a different platform the sizes do not change.

### What is Struct Padding?

Struct padding is memory inserted by the compiler between struct members to satisfy alignment requirements. The Cortex-M4 requires that a 32-bit value be stored at a 4-byte aligned address, a 16-bit value at a 2-byte aligned address, and an 8-bit value at any address. When struct members have different sizes the compiler inserts invisible padding bytes between them to ensure each member lands on an aligned boundary.

This means the size of a struct is not simply the sum of its members. It depends on the order and sizes of those members, and it can vary between compilers and architectures. A struct that is 8 bytes on one platform may be 12 bytes on another. If you are sending a struct over UART, SPI, or I2C to another device, or storing it in flash, or comparing struct layouts between a microcontroller and a host application, unexpected padding will corrupt your data silently.

Understanding padding is essential for anyone writing sensor drivers, communication protocols, or data logging firmware.

### Why is This Relevant to Robotics?

A robot aggregates data from many sensors simultaneously -- distance, temperature, heading, battery voltage, motor current, joint angles. All of that data needs to be stored, processed, and often transmitted. In a multi-robot system or a robot with a companion computer, sensor structs are frequently sent over serial links between devices.

If the struct on the microcontroller and the struct on the receiving computer have different layouts due to padding differences, every field after the first misaligned member will be read incorrectly. The robot may report the wrong position, the companion computer may command the wrong velocity. The entire system behaves erratically with no obvious error.

Choosing fixed-width types and understanding struct layout is not pedantry. It is the foundation of correct data exchange in any system where more than one processor is involved -- which describes almost every serious robotics platform.

### What This Experiment Demonstrates

This experiment builds a sensor aggregation struct three ways -- using lazy platform-dependent types, using fixed-width types in a naive member order, and using fixed-width types in an optimised member order -- and measures the size and layout of each using sizeof and offsetof. You will then transmit a struct over UART and reconstruct it on the receiving end to see what padding does to a live data exchange. By the end you will be able to:

- Choose appropriate fixed-width types for sensor data based on the actual range of each measurement. Predict struct size by accounting for alignment padding.
- Eliminate padding waste by ordering struct members from largest to smallest.
- Use **attribute**((packed)) correctly and understand its tradeoffs. Explain why struct layout matters for cross-device communication in robotics.

### Part 3 -- Struct Definitions

Place all three struct definitions in the private types section of main.c:

```c
/* USER CODE BEGIN PTD */

// Version 1 -- lazy types
// Uses int throughout regardless of actual data range
// Size and layout are platform-dependent
typedef struct {
    int distance;       // Intended range: 0-255 cm.   int = 4 bytes on ARM
    int temperature;    // Intended range: -40 to 85.  int = 4 bytes on ARM
    int heading;        // Intended range: 0-360.       int = 4 bytes on ARM
    int battery;        // Intended range: 0-100%.      int = 4 bytes on ARM
    int status;         // Boolean flag: 0 or 1.        int = 4 bytes on ARM
} SensorReport_Lazy;

// Version 2 -- fixed-width types, naive member order
// Each member is sized correctly for its data range
// but ordering is not optimised -- may still contain padding
typedef struct {
    uint8_t  status;        // 1 byte  -- boolean flag
    uint16_t heading;       // 2 bytes -- 0 to 360 degrees
    uint8_t  distance;      // 1 byte  -- 0 to 255 cm
    int8_t   temperature;   // 1 byte  -- -40 to 85 degrees C
    uint8_t  battery;       // 1 byte  -- 0 to 100 percent
} SensorReport_Naive;

// Version 3 -- fixed-width types, optimised member order
// Members ordered largest to smallest to eliminate padding
typedef struct {
    uint16_t heading;       // 2 bytes -- 0 to 360 degrees
    uint8_t  distance;      // 1 byte  -- 0 to 255 cm
    int8_t   temperature;   // 1 byte  -- -40 to 85 degrees C
    uint8_t  battery;       // 1 byte  -- 0 to 100 percent
    uint8_t  status;        // 1 byte  -- boolean flag
} SensorReport_Optimised;

// Version 4 -- packed struct
// Forces no padding -- use only when exact wire layout is required
typedef struct __attribute__((packed)) {
    uint8_t  status;
    uint16_t heading;
    uint8_t  distance;
    int8_t   temperature;
    uint8_t  battery;
} SensorReport_Packed;

/* USER CODE END PTD */
```

Section A is the headline result. The lazy struct costs 20 bytes for data that genuinely only needs 6. In a ring buffer of 32 sensor reports that is 640 bytes versus 192 bytes -- a difference that matters on a device with 192KB of SRAM being shared between your application, stack, and heap.

Section B shows where the padding lives in the naive version. The uint8_t status at offset 0 forces a padding byte before heading at offset 2 because uint16_t requires 2-byte alignment. The optimised version puts heading first and eliminates that gap entirely.

Section D is the most important result for robotics. Every field after the first misaligned member is read incorrectly when the two sides use different struct layouts. A heading of 270 becomes 1. A temperature of -10 becomes 1. A battery of 85 becomes 246. This is not a subtle off-by-one error -- it is total data corruption, and it happens silently with no error message of any kind.

Section E shows the packed struct eliminating the problem entirely. When both sides agree on the exact byte layout there is no ambiguity and every field is received correctly.

### Part 7 -- When to Use Each Approach

Use fixed-width types with optimised member ordering as your default for all sensor structs. This gives you the smallest possible size without any alignment penalty and the fastest possible member access since all reads are naturally aligned.

Use **attribute**((packed)) only when you need to match an exact wire format -- for example when implementing a documented sensor protocol, a radio packet format, or a binary log format that must be read by an external tool. Packed structs carry a performance cost on Cortex-M4 because unaligned multi-byte reads require additional instructions, and they cannot be safely used with DMA on all STM32 peripherals. Never use packed as a lazy substitute for thinking about member ordering.

Never use platform-dependent types like int, long, or unsigned in sensor structs or any data structure that crosses a process, device, or architecture boundary. Reserve int for loop counters and throwaway temporaries where portability does not matter.

### What You Should Take Away

The lazy struct wastes 14 bytes compared to the fixed-width version for exactly the same data. Multiplied across large arrays or ring buffers this becomes a significant fraction of available SRAM. Member ordering determines whether padding appears -- placing larger members first eliminates gaps and produces the smallest naturally aligned struct. Padding in a transmitted struct silently corrupts every field after the first misalignment on the receiving end. This is one of the most common sources of communication bugs in multi-device robotics systems. The packed attribute eliminates padding for wire transmission but carries an access cost -- it is a tool for protocol matching, not a general-purpose memory optimiser. Fixed-width types make the intended range of every variable explicit, make struct sizes predictable, and make your code portable across every embedded target you will ever use. Using them by default costs nothing and prevents an entire class of subtle, hard-to-diagnose bugs.
