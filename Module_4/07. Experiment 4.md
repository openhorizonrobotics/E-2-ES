
## Dangling Pointer from a Returned Local Buffer

"Never return a pointer to a local variable."

### What is a Dangling Pointer?

A dangling pointer is a pointer that holds an address which no longer contains valid data. The pointer itself is not broken, it still holds a memory address and you can dereference it without an immediate crash. The problem is that the memory it points to has been reclaimed and may have been overwritten by something else. Reading from it gives you garbage. Writing to it corrupts whatever currently owns that memory. Neither situation produces an obvious error message. Both situations produce bugs that are intermittent, hard to reproduce, and deeply confusing to debug.

In embedded C the most common way to create a dangling pointer is to return the address of a local variable from a function. This is a mistake that looks completely reasonable when you write it, compiles without error, and often appears to work during testing. Understanding why it is wrong requires understanding what actually happens to local variables when a function returns.

### The Stack Frame Lifecycle

Every time a function is called, the processor carves out a region of the stack called a stack frame. This frame holds the function's local variables, its saved registers, and the return address that tells the CPU where to go when the function finishes. The frame is created when the function is called and destroyed when the function returns; the stack pointer moves back up and that memory is considered free.

The key word is considered. The memory is not zeroed. The values are not erased. They sit in SRAM exactly as they were until something else comes along and overwrites them. If you hold a pointer to that region and read from it immediately after the function returns, you will often see the original values still there because nothing has had a chance to overwrite them yet. This is what makes dangling pointer bugs so treacherous. They appear to work.

Call another function after the one that returned the dangling pointer and the frame gets reused. Now the memory your pointer references contains the local variables of a completely different function. Read from it and you get that function's data. Write to it and you corrupt that function's stack frame. The behaviour is undefined and non-deterministic, it depends entirely on what else happens to use that region of the stack.

### Why is This Relevant to Robotics?

Robot arm firmware is a natural place for this bug to appear. A typical inverse kinematics pipeline computes joint angles inside a function, stores them in a local array, and needs to pass them back to the caller for use by the motor controllers. The instinct -- especially for a programmer coming from higher-level languages, is to return a pointer to that local array. It is concise, it avoids an extra parameter, and it appears to work in simple test cases.

In a real system the motor controller, the safety monitor, and the telemetry logger all run between the kinematics call and the point where the angles are consumed. Each of those functions uses the stack. By the time the angles are read, the memory they occupied has been overwritten multiple times. The motor controller receives garbage angles. The arm moves to the wrong position. The safety monitor may or may not catch it depending on timing.

This is not a hypothetical failure mode. It is a recurring class of bug in embedded C that appears specifically in robotics pipelines where computed values are passed between processing stages.

### What This Experiment Demonstrates

This experiment constructs a minimal version of this bug on the STM32F4 and makes the corruption visible over UART. You will see the correct values immediately after the bad function returns, then watch them get overwritten as subsequent function calls reuse the stack frame. By the end you will be able to:

Identify the dangling pointer pattern and explain why it is wrong at the stack level. Observe the corruption window -- the brief period where the values appear correct before being overwritten. Apply three correct alternatives: static allocation, caller-provided buffer, and heap allocation with explicit ownership.

### Part 1 — Project Setup

- Open STM32CubeIDE, go to File, New, STM32 Project.
- Search for STM32F407VG, select the Discovery board, name the project (e.g. Exp4_DanglingPointer), and click Finish.
- In the .ioc configurator, enable USART2 under Connectivity. Set mode to Asynchronous, baud rate 115200, 8N1. Under System Core, SYS, set Debug to Serial Wire. Save and generate code.
- Set optimisation to -O0 for this experiment. The dangling pointer bug is a memory correctness issue, not a compiler optimisation issue, so -O0 gives the clearest and most predictable UART output. Right-click the project, open Properties, navigate to C/C++ Build, Settings, MCU GCC Compiler, Optimization. Set to -O0. Click Apply and Close.

### Part 3 — The Experiment Code

The code is structured in three sections. Section A demonstrates the bug and makes the corruption visible. Section B shows the corruption getting worse when more stack activity happens between the bad return and the read. Section C shows all three correct fixes side by side.

Place all function definitions before main() in the user code section:

```c
/* USER CODE BEGIN PFP */

// ---------------------------------------------------------------

// SECTION A — The bug: returning a pointer to a local array

// ---------------------------------------------------------------

void compute_angles_bad(void) {
volatile float angles[3];
angles[0] = 45.0f;
angles[1] = 90.0f;
angles[2] = 135.0f;

// Capture the address before the frame is destroyed
captured_address = (uint32_t)angles;
printf("[bad] inside function, address: 0x%08lX\r\n", captured_address);
printf("[bad] angles[0] = %.2f\r\n", angles[0]);
printf("[bad] angles[1] = %.2f\r\n", angles[1]);
printf("[bad] angles[2] = %.2f\r\n", angles[2]);
}

  
// ---------------------------------------------------------------
// SECTION B — A function that reuses the stack after the bad return
// ---------------------------------------------------------------

// This function does nothing useful — it exists purely to overwrite
// the stack region that compute_angles_bad() previously occupied

void overwrite_stack(void) {
volatile float trash[16];
for (int i = 0; i < 16; i++) {
trash[i] = i * 1.1f;
}
}

// ---------------------------------------------------------------
// SECTION C — Three correct alternatives
// ---------------------------------------------------------------

// Fix 1: Static local — lives in .bss, not on the stack
// Safe to return a pointer to because it is never destroyed

float* compute_angles_static(void) {
static float angles[3];
angles[0] = 45.0f;
angles[1] = 90.0f;
angles[2] = 135.0f;
return angles;
}

// Fix 2: Caller-provided buffer — caller owns the memory, no ambiguity
// This is the preferred pattern in production embedded C

void compute_angles_caller(float *out, uint8_t len) {
if (out == NULL || len < 3) return;
out[0] = 45.0f;
out[1] = 90.0f;
out[2] = 135.0f;
}

// Fix 3: Heap allocation — caller is responsible for freeing
// Use only when lifetime is genuinely dynamic

float* compute_angles_heap(void) {
float *angles = (float*)malloc(3 * sizeof(float));
if (angles == NULL) return NULL;
angles[0] = 45.0f;
angles[1] = 90.0f;
angles[2] = 135.0f;
return angles;
}

/* USER CODE END PFP */
```

Section A is the deceptive case. The values read back correctly because nothing has touched that stack region yet. This is exactly why the bug survives testing — in a simple program with no subsequent function calls it always looks fine.

Section B is where the corruption becomes visible. overwrite_stack() reuses the exact same stack region and replaces the joint angles with its own local values. The dangling pointer now reads back 1.10, 2.20, 3.30 instead of 45.00, 90.00, 135.00. In a real robot arm this would send each joint to the wrong position.

Section C shows all three fixes producing the correct values reliably.

### Part 5 — Choosing the Right Fix

Each of the three correct approaches suits a different situation. Understanding when to use each one is as important as knowing the bug itself.

Fix 1: static local is the simplest change and requires no modification to the function signature. However it has one important limitation: there is only ever one instance of the array. If the function is called again before the caller has finished using the previous result, the values will be overwritten. This makes it unsuitable for reentrant functions or interrupt-driven pipelines where the function might be called from both the main loop and an ISR.

Fix 2: caller-provided buffer is the preferred pattern in production embedded C. The caller declares the array, owns its lifetime, and passes it in. There is no ambiguity about who is responsible for the memory, no heap involved, and no reentrancy problem. The function signature makes the ownership explicit. This is the approach you will see in well-written embedded libraries and HAL drivers.

Fix 3: heap allocation works and makes ownership explicit through the return value, but it introduces all the failure modes covered in Experiment 3. The caller must remember to call free, the allocation can fail, and in a long-running system repeated allocation and deallocation of small fixed-size buffers will fragment the heap. For a buffer whose size is known at compile time and whose lifetime is a single pipeline stage, this is the least appropriate of the three fixes. It is worth knowing but should not be your default.

### What You Should Take Away

Section A shows that the dangling pointer often appears to work immediately after the bad return. The stack memory is not zeroed, the values sit there until something overwrites them. Section B shows that any subsequent function call can overwrite that memory, turning correct-looking values into garbage. The corruption is non-deterministic: it depends entirely on what else uses the stack between the bad return and the read. GCC will warn about this pattern if you compile with -Wall -Wextra. Always build with warnings enabled and treat this warning as an error. The caller-provided buffer pattern eliminates the bug entirely, makes ownership explicit, and adds no runtime cost. It is the correct default for passing computed arrays out of functions in embedded C. Static locals are a quick fix but are not reentrant, use them only when you are certain the function will never be called concurrently from multiple contexts. Heap allocation solves the lifetime problem but introduces allocation failure and leak risk; avoid it for fixed-size buffers with a known lifetime.

The broader principle is this: in C, memory does not manage itself. Every array and every buffer has a lifetime and an owner, and it is your job to make both explicit. Returning a pointer to a local variable is the language letting you express something that is physically impossible -- the memory is gone the moment the function returns. The compiler may warn you. The hardware will not protect you. The bug will wait until the worst possible moment to appear.
