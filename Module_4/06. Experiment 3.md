# Static vs Dynamic Allocation for a Sensor Ring Buffer

*"Prefer static allocation in long-running embedded systems!"*

### Memory Allocation in Embedded C

Every variable in your program needs memory. In embedded C there are three places it can come from, and the choice between them is not just a matter of style, it has direct consequences for reliability, determinism, and the long-term behaviour of your firmware.

The stack gives you memory automatically when a function is called and takes it back when the function returns. It is fast and self-managing but finite and short-lived. The heap gives you memory on demand at runtime via malloc and calloc, and you hold onto it until you explicitly release it with free. Static allocation gives you memory that is reserved at compile time, lives for the entire duration of the program, and costs nothing at runtime to acquire or release.

On a desktop machine the heap is enormous and the operating system manages it for you. On a microcontroller with 192KB of SRAM, a fixed heap region of perhaps 8KB, no memory protection unit in the default configuration, and a program that must run continuously for hours or days without a reset, the heap is a liability. This experiment shows you exactly why.

### What is a Ring Buffer?

A ring buffer (also called a circular buffer), is one of the most commonly used data structures in embedded systems. It is a fixed-size array that behaves as if its two ends are connected, allowing data to be written and read continuously without ever running out of space, as long as the reader keeps up with the writer.

It works by maintaining two indices: a head, which tracks where the next write will go, and a tail, which tracks where the next read will come from. When either index reaches the end of the array it wraps back to zero. The buffer is full when the head is one position behind the tail. It is empty when the head and tail are equal.

```
Index:  0    1    2    3    4    5    6    7
       [120][134][98 ][   ][   ][   ][109][115]
                  ^                   ^
                 head               tail
        (next write here)     (next read here)
```

Ring buffers appear everywhere in robotics firmware: buffering UART bytes from a GPS module, storing ADC samples from a current sensor, queuing encoder ticks for an odometry pipeline, or holding IMU readings between filter updates. They are the natural solution whenever a producer generates data at a different rate than the consumer processes it.

### Why Does Allocation Strategy Matter for a Ring Buffer?

A ring buffer needs a fixed-size backing array. The question is where that array lives, and that question has more consequences than it first appears.

If you allocate the backing array with malloc, several things become true. The allocation can fail, returning NULL, if the heap is exhausted or fragmented. The allocation happens at runtime, introducing a small non-deterministic delay. Every allocation carries hidden overhead; newlib's malloc prepends a block header to every allocation, so malloc(256) actually consumes more than 256 bytes of heap. And if you ever forget to call free, or if the code path that calls free is never reached due to an error condition, that memory is gone for the rest of the session.

If you allocate the backing array statically, none of those things are true. The memory is reserved at compile time, it cannot fail, and it has no runtime overhead. There's no worry about leaks either. The size is visible in your map file. It exists from startup to shutdown with a fixed, predictable address.

For a data structure whose size you know in advance (and a ring buffer always has a known fixed size) there is no compelling reason to use the heap. This experiment demonstrates both approaches side by side so you can see the difference in behaviour, memory cost, and failure handling on the STM32F4.

### Why is This Relevant to Robotics?

Consider a differential drive robot maintaining odometry: a running estimate of its position based on wheel encoder ticks. The encoder ISR fires hundreds or thousands of times per second, producing a stream of tick counts that must be buffered and consumed by a position update loop running at a lower rate.

This buffer must work correctly from the moment the robot powers on to the moment it powers off. It will never be freed and reallocated. Its size is known at design time. It will be written by an ISR and read by the main loop. There is no scenario in which dynamic allocation adds value here and several scenarios in which it introduces risk.

This is true of the majority of buffers in robotics firmware. Sensor pipelines, command queues, telemetry logs; almost all of them have a known fixed size and a lifetime equal to the program itself. Defaulting to static allocation for these structures is not a limitation. It is the correct engineering choice.

### What This Experiment Demonstrates

This experiment implements the same encoder tick ring buffer twice, once using dynamic allocation with malloc, and once using static allocation. Both implementations are exercised under identical conditions. You will observe how the dynamic version behaves when allocation fails, how much heap it actually consumes beyond the requested size, and how the static version eliminates every failure mode the dynamic version introduces. By the end you will be able to:

- Implement a ring buffer using both dynamic and static allocation strategies.
- Deliberately exhaust the heap to trigger and handle a malloc failure gracefully.
- Measure the true heap cost of a dynamic allocation including allocator overhead. 
- Explain why static allocation is the correct default for fixed-size buffers in long-running embedded systems.

### Part 3 — Ring Buffer Implementation

This section defines both versions of the ring buffer. Place the entire block in the private types section of main.c:

```c
/* USER CODE BEGIN PTD */

#define BUFFER_CAPACITY 32    // Number of encoder tick samples the buffer holds

// Shared struct layout used by both versions.
// head  — index of the next write position
// tail  — index of the next read position
// count — number of samples currently stored
typedef struct {
    int32_t *data;            // Pointer to the backing array
    uint8_t  head;
    uint8_t  tail;
    uint8_t  count;
} RingBuffer;

/* USER CODE END PTD */
```

Now define the static backing array and the static buffer instance. Place these in the private variables section:

```c
/* USER CODE BEGIN PV */

// Static version — backing array reserved at compile time in .bss
static int32_t        static_data[BUFFER_CAPACITY];
static RingBuffer     static_buf = { static_data, 0, 0, 0 };

/* USER CODE END PV */
```

Now define the ring buffer operations. Place these before main():

```c
/* USER CODE BEGIN PFP */

// Write a tick value into the buffer.
// Returns 1 on success, 0 if the buffer is full.
uint8_t rb_write(RingBuffer *rb, int32_t value) {
    if (rb->count == BUFFER_CAPACITY) return 0;
    rb->data[rb->head] = value;
    rb->head = (rb->head + 1) % BUFFER_CAPACITY;
    rb->count++;
    return 1;
}

// Read the oldest tick value from the buffer.
// Returns 1 on success, 0 if the buffer is empty.
uint8_t rb_read(RingBuffer *rb, int32_t *out) {
    if (rb->count == 0) return 0;
    *out = rb->data[rb->tail];
    rb->tail = (rb->tail + 1) % BUFFER_CAPACITY;
    rb->count--;
    return 1;
}

// Allocate a new ring buffer on the heap.
// Returns NULL if either allocation fails.
RingBuffer* rb_create_dynamic(void) {
    RingBuffer *rb = (RingBuffer*)malloc(sizeof(RingBuffer));
    if (rb == NULL) {
        printf("[dynamic] RingBuffer struct alloc FAILED — heap exhausted\r\n");
        return NULL;
    }
    rb->data = (int32_t*)malloc(BUFFER_CAPACITY * sizeof(int32_t));
    if (rb->data == NULL) {
        printf("[dynamic] Backing array alloc FAILED — heap exhausted\r\n");
        free(rb);
        return NULL;
    }
    rb->head  = 0;
    rb->tail  = 0;
    rb->count = 0;
    return rb;
}

// Free a dynamically allocated ring buffer.
void rb_destroy_dynamic(RingBuffer *rb) {
    if (rb == NULL) return;
    free(rb->data);
    free(rb);
}

/* USER CODE END PFP */
```

### Part 4 — The Experiment Code

Place the entire experiment inside the USER CODE BEGIN 2 section in main():

```c
/* USER CODE BEGIN 2 */

printf("\r\n=== Experiment 3: Static vs Dynamic Ring Buffer ===\r\n\n");

// ---------------------------------------------------------------
// SECTION A — Dynamic allocation: normal case
// ---------------------------------------------------------------

printf("--- Section A: Dynamic allocation (normal) ---\r\n");

RingBuffer *dyn_buf = rb_create_dynamic();

if (dyn_buf != NULL) {

    // Simulate encoder ISR writing tick values into the buffer
    for (int i = 0; i < 8; i++) {
        int32_t tick = (i + 1) * 10;    // Simulated tick: 10, 20, 30 ...
        rb_write(dyn_buf, tick);
        printf("[dynamic] wrote tick: %ld\r\n", tick);
    }

    // Odometry loop reading ticks back out
    int32_t val;
    printf("\r\n[dynamic] reading back:\r\n");
    while (rb_read(dyn_buf, &val)) {
        printf("[dynamic] read tick: %ld\r\n", val);
    }

    // Report actual heap cost
    // sizeof(RingBuffer) + sizeof backing array + two malloc block headers
    // newlib-nano prepends 8 bytes of overhead per allocation on Cortex-M
    size_t requested = sizeof(RingBuffer) + (BUFFER_CAPACITY * sizeof(int32_t));
    size_t actual    = requested + 16;    // Two 8-byte block headers
    printf("\r\n[dynamic] bytes requested: %u\r\n", requested);
    printf("[dynamic] bytes consumed (with headers): ~%u\r\n", actual);

    rb_destroy_dynamic(dyn_buf);
    printf("[dynamic] buffer freed\r\n\n");
}

// ---------------------------------------------------------------
// SECTION B — Dynamic allocation: heap exhaustion
// ---------------------------------------------------------------

printf("--- Section B: Dynamic allocation (heap exhaustion) ---\r\n");

// Consume the heap deliberately with a series of small allocations
// until malloc begins failing
RingBuffer *victims[16];
int allocated = 0;

for (int i = 0; i < 16; i++) {
    victims[i] = rb_create_dynamic();
    if (victims[i] == NULL) {
        printf("[dynamic] heap exhausted after %d successful allocations\r\n",
               allocated);
        break;
    }
    allocated++;
    printf("[dynamic] allocation %d succeeded\r\n", allocated);
}

// Try to allocate one more buffer after exhaustion
printf("\r\n[dynamic] attempting one more allocation on exhausted heap...\r\n");
RingBuffer *failed = rb_create_dynamic();
if (failed == NULL) {
    printf("[dynamic] correctly returned NULL — failure handled\r\n\n");
}

// Clean up everything that was successfully allocated
for (int i = 0; i < allocated; i++) {
    rb_destroy_dynamic(victims[i]);
}
printf("[dynamic] all successful allocations freed\r\n\n");

// ---------------------------------------------------------------
// SECTION C — Static allocation: same workload, no failure modes
// ---------------------------------------------------------------

printf("--- Section C: Static allocation ---\r\n");

// The static buffer is always ready — no allocation, no failure mode
printf("[static] buffer address: 0x%08lX\r\n", (uint32_t)&static_buf);
printf("[static] data array address: 0x%08lX\r\n", (uint32_t)static_data);
printf("[static] sizeof RingBuffer: %u bytes\r\n", sizeof(RingBuffer));
printf("[static] sizeof backing array: %u bytes\r\n",
       BUFFER_CAPACITY * sizeof(int32_t));
printf("[static] total footprint: %u bytes (compile-time, no overhead)\r\n\n",
       sizeof(RingBuffer) + BUFFER_CAPACITY * sizeof(int32_t));

// Same encoder tick workload as Section A
for (int i = 0; i < 8; i++) {
    int32_t tick = (i + 1) * 10;
    rb_write(&static_buf, tick);
    printf("[static] wrote tick: %ld\r\n", tick);
}

int32_t val;
printf("\r\n[static] reading back:\r\n");
while (rb_read(&static_buf, &val)) {
    printf("[static] read tick: %ld\r\n", val);
}

printf("\r\n=== Experiment complete ===\r\n");

/* USER CODE END 2 */
```

### The key observations are:

Section A shows that the dynamic version works correctly under normal conditions but consumes more memory than requested due to allocator overhead. Section B shows that heap exhaustion is a real event on this hardware, it happens after only a handful of allocations, and that without NULL checks the program would crash silently. Section C shows that the static version produces identical functional output, has a lower and precisely known memory footprint, and has no failure modes whatsoever.

### Part 6 — Inspecting Memory Layout with the Map File

After building the project, open the generated map file. In STM32CubeIDE, find it in your project's Debug folder: Exp3_RingBuffer.map.

Search for static_data and static_buf. You will find entries like:

```
.bss.static_data    0x20000000    0x80
.bss.static_buf     0x20000080    0x08
```

This confirms that both the backing array and the struct are placed in .bss in SRAM at fixed addresses determined at link time. Their addresses are known before the program runs. Their sizes are exact. There is no runtime overhead and no possibility of placement failure.

Now search for any symbol associated with heap. You will find the heap region defined by the linker but no symbols for the dynamically allocated buffers, because their addresses are not known until runtime and therefore cannot appear in the map file. This is the tradeoff: flexibility at runtime in exchange for visibility at compile time.

### Part 7 — The Fix in Production Code

In production robotics firmware the static version is not just acceptable, it is the standard. The correct pattern for a fixed-size ring buffer in a long-running embedded system is:

```c
// Define capacity as a named constant — visible, auditable, changeable
#define ENCODER_BUF_CAPACITY  32

// Backing array in .bss — fixed address, zero runtime cost
static int32_t encoder_data[ENCODER_BUF_CAPACITY];

// Struct initialised at compile time — no init function required
static RingBuffer encoder_buf = { encoder_data, 0, 0, 0 };
```

This pattern has several properties that matter in production. The buffer is ready at startup with no initialisation call required. Its memory footprint appears in the map file and can be audited. It cannot fail at runtime. It cannot leak. Its address is stable across resets, which matters when using a debugger or a logic analyser to monitor live buffer state.

Reserve malloc for the rare cases where size is genuinely unknown at compile time or where objects need to be created and destroyed dynamically at runtime. For everything else, and in robotics firmware that is most things, static allocation is safer, faster, and more transparent.
