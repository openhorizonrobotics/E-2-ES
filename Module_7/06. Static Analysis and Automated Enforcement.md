Writing good code is necessary but not sufficient. Even experienced engineers make mistakes, and a coding standard that lives only in a document and in the heads of team members will be violated, not out of negligence, but because humans are inconsistent under pressure, during code review fatigue, and across a codebase that spans hundreds of files and multiple contributors. Static analysis is the discipline of having a tool examine your source code before it runs and identify violations automatically.

This chapter covers what static analysis tools do, which ones matter in a robotics context, and how to integrate them into a development workflow so that violations are caught as early as possible, ideally before code is ever reviewed by a human.

### What Static Analysis Actually Does

A compiler already performs a limited form of static analysis. When GCC warns you about an unused variable or a comparison between signed and unsigned types, it is applying a simple rule to the code's structure without executing it. Static analysis tools extend this idea substantially further. They build a detailed model of your program: tracking data flow, control flow, pointer aliases, and type information -- and apply a much larger set of rules to that model.

The distinction between a compiler warning and a static analysis finding is one of depth. A compiler sees one translation unit at a time and applies relatively simple pattern-matching rules. A static analysis tool can reason across function boundaries, follow values through multiple assignments, and detect conditions that only arise along specific execution paths. This allows it to catch a class of bugs that no compiler warning will ever find.

Consider a function that conditionally initializes a pointer and then uses it:

```c
#include <stdint.h>

void process_sensor_data(uint8_t sensor_id) {
    uint8_t* data_ptr;

    if (sensor_id < MAX_SENSORS) {
        data_ptr = get_sensor_buffer(sensor_id);
    }

    /* Static analysis flags this: data_ptr may be uninitialized
       if sensor_id >= MAX_SENSORS */
    uint8_t first_byte = *data_ptr;
    process_byte(first_byte);
}
```

A compiler with `-Wall` enabled may or may not warn about this, depending on the complexity of the control flow. A static analysis tool will almost always find it, because it explicitly models the path where the `if` condition is false and traces what `data_ptr` contains on that path.

This is the fundamental value proposition of static analysis: it finds bugs along paths that testing may never exercise, before the code is deployed to hardware.

### Compiler Warnings as the First Line of Defense

Before reaching for a dedicated static analysis tool, the compiler itself should be configured to report as many problems as it can. GCC and Clang both support extensive warning flags, and enabling them costs nothing beyond a slightly longer build output to review.

The flags that every production robotics project should enable are:

```
-Wall -Wextra -Wpedantic -Wshadow -Wconversion -Wsign-conversion -Werror
```

`-Wall` enables the warnings GCC considers most important -- misleadingly, it does not enable all warnings, just the ones with a low false positive rate. `-Wextra` adds a second tier of warnings that are slightly more opinionated. `-Wpedantic` enforces strict conformance to the language standard, catching extensions that work on your compiler but may not be portable. `-Wshadow` catches variable declarations that shadow an outer-scope variable with the same name, which is a common source of subtle bugs in C++ where a local variable silently hides a class member. `-Wconversion` and `-Wsign-conversion` flag implicit numeric conversions that may change a value, directly corresponding to the MISRA type safety rules from Chapter 2.

`-Werror` promotes all warnings to errors, causing the build to fail when any warning is present. This is the most important flag of the set. A warning that does not fail the build is a warning that gets ignored. Teams that accumulate hundreds of suppressed warnings in their build output lose the signal entirely -- new warnings from real bugs are invisible in the noise.

In STM32CubeIDE, these flags are added in the project properties under C/C++ Build → Settings → Tool Settings → MCU GCC Compiler → Miscellaneous, in the Other flags field. Add them to the Debug configuration during development, and ensure the Release configuration also builds cleanly with all warnings enabled.

### cppcheck: Open Source MISRA Checking

`cppcheck` is an open source static analysis tool with specific support for MISRA C:2012 checking. It is not a commercial tool and its MISRA coverage is not complete, but it catches a substantial fraction of the most common violations and is easy to integrate into both a local development workflow and a CI pipeline. For a team that cannot afford a commercial analyzer, it is the right starting point.

STM32CubeIDE includes `cppcheck` integration. To enable it, right-click your project, select Properties, navigate to C/C++ General → Code Analysis, and enable the Cppcheck option. Under the configuration, you can specify which rule sets to check against.

For command-line use (which is what you will need for CI integration), `cppcheck` is invoked like this:

```
cppcheck \
  --enable=all \
  --addon=misra \
  --suppressions-list=misra_suppressions.txt \
  --error-exitcode=1 \
  --xml \
  --xml-version=2 \
  -I Core/Inc \
  Core/Src/
```

Walking through each flag: `--enable=all` enables all of cppcheck's own checks in addition to the MISRA addon. `--addon=misra` activates the MISRA C:2012 checker. `--suppressions-list` points to a file containing documented suppressions: the equivalent of a deviation register for this tool. `--error-exitcode=1` causes cppcheck to return a non-zero exit code when it finds violations, which is what CI systems use to determine whether a step has passed or failed. `--xml` produces machine-readable output suitable for ingestion by CI reporting tools.

The suppressions file deserves attention. It is not a way to make violations disappear silently. Each suppression should be documented with the same rigor as a formal deviation:

```
// misra_suppressions.txt
// Rule 11.5: Cast from void* in HAL interface - required by STM32 HAL design
misra-c2012-11.5:Core/Src/hal_interface.c

// Rule 21.3: malloc used in startup phase only, before scheduler starts
misra-c2012-21.3:Core/Src/startup_alloc.c
```

Running cppcheck against the fault diagnostics project from Chapter 4 gives a useful demonstration. Create a small C file with deliberate violations:

```c
/* test_violations.c */
#include <stdint.h>
#include <stdlib.h>

void demonstrate_violations(void) {
    /* MISRA C:2012 Rule 9.1: variable used before initialization */
    int16_t value;
    int16_t result = value + 1;

    /* MISRA C:2012 Rule 21.3: dynamic memory allocation */
    uint8_t* buffer = (uint8_t*)malloc(64);

    /* MISRA C:2012 Rule 10.4: mixed signed/unsigned arithmetic */
    uint32_t count = 10U;
    int32_t  offset = -5;
    int32_t  combined = count + offset;

    (void)result;
    (void)buffer;
    (void)combined;
}
```

Running cppcheck against this file produces output identifying each violation by rule number, file, and line. The rule number maps directly back to the MISRA C:2012 handbook and to the explanations in Chapter 2, which is why understanding the standards at the level covered there is practically useful. You can read a cppcheck finding and immediately understand what it is telling you without looking anything up.

### clang-tidy: Modern C++ Analysis

While cppcheck is oriented toward C and MISRA compliance, `clang-tidy` is the tool of choice for modern C++ analysis. It is built on the Clang compiler infrastructure, which means it has access to the same detailed semantic model that Clang uses for compilation. This allows it to reason about C++ templates, type deduction, and ownership semantics in ways that simpler tools cannot.

`clang-tidy` organizes its checks into named categories. The ones most relevant to robotics are:

The `cppcoreguidelines` checks enforce the C++ Core Guidelines, a set of best practices maintained by Bjarne Stroustrup and Herb Sutter that overlaps substantially with AUTOSAR C++14 guidelines. These checks catch things like raw pointer ownership issues, missing `[[nodiscard]]` on functions that return error codes, and unsafe use of C-style casts.

The `modernize` checks identify code that is written in an older C++ style when a safer modern equivalent exists. `modernize-use-nullptr` flags uses of `NULL` or `0` as null pointer constants that should be `nullptr`. `modernize-use-override` flags virtual function overrides that are missing the `override` keyword.

The `bugprone` checks identify patterns that are not necessarily standard violations but are statistically likely to be bugs. `bugprone-integer-division` catches integer division where a floating point result was probably intended. `bugprone-signed-char-misuse` catches the common mistake of treating `char` as a signed type when processing byte data.

A `.clang-tidy` configuration file placed at the root of your project specifies which checks to enable:

```yaml
Checks: >
  cppcoreguidelines-*,
  modernize-use-nullptr,
  modernize-use-override,
  modernize-use-default-member-init,
  bugprone-*,
  -bugprone-easily-swappable-parameters,
  readability-identifier-naming

WarningsAsErrors: '*'

CheckOptions:
  - key: readability-identifier-naming.ClassCase
    value: CamelCase
  - key: readability-identifier-naming.FunctionCase
    value: lower_case
  - key: readability-identifier-naming.ConstantCase
    value: UPPER_CASE
```

The `-bugprone-easily-swappable-parameters` suppression illustrates a common pattern: some checks have a high false positive rate for legitimate code and are better disabled globally. The `readability-identifier-naming` checks enforce a naming convention consistently across the entire codebase: something that is tedious to enforce manually in code review but trivial to automate.

Running `clang-tidy` against a file requires a compilation database -- a JSON file that records the exact compiler flags used for each translation unit. CMake generates this automatically with `-DCMAKE_EXPORT_COMPILE_COMMANDS=ON`. For STM32CubeIDE projects, generating a compilation database requires a small additional step: the `bear` tool can intercept the build process and generate `compile_commands.json` automatically.

```bash
# Generate compilation database from a CubeIDE build
bear -- make -C /path/to/project/Debug

# Run clang-tidy against all source files
clang-tidy \
  -p compile_commands.json \
  Core/Src/*.c \
  Core/Src/*.cpp
```

### Commercial Tools: PC-lint and Polyspace

Open source tools cover a large fraction of practical static analysis needs, but commercial tools exist at a higher tier of thoroughness and certification support. Two are worth knowing by name.

PC-lint Plus, from Gimpel Software ([https://www.gimpel.com](https://www.gimpel.com)), has been the industry standard for MISRA-compliant C and C++ analysis for decades. Its MISRA C:2012 and MISRA C++:2008 coverage is more complete than cppcheck's, its false positive rate is lower, and it produces the kind of detailed analysis reports that safety auditors expect. For a project targeting ASIL B or above under ISO 26262, PC-lint Plus or an equivalent commercial tool is effectively required.

Polyspace, from MathWorks ([https://www.mathworks.com/products/polyspace.html](https://www.mathworks.com/products/polyspace.html)), takes a different approach. Rather than pattern matching against rules, it uses abstract interpretation -- a formal mathematical technique that over-approximates all possible runtime values of every variable in the program. This allows it to prove the absence of certain runtime errors, not just flag suspicious patterns. Polyspace can produce a formal proof that a piece of code never dereferences a null pointer or never overflows a signed integer, which is a qualitatively stronger statement than "no violation was found." For ASIL D software, this kind of formal evidence is often required.

The practical reality for most robotics teams is that cppcheck and clang-tidy are the right starting point, and commercial tools are adopted when the project's safety requirements demand them. Knowing they exist and understanding what they offer is enough for now.

### Integrating Static Analysis into CI

A static analysis tool that is only run occasionally, manually, by developers who remember to run it, is a static analysis tool that does not actually enforce anything. The only configuration that provides real enforcement is one where the tool runs automatically on every proposed code change and blocks merging if violations are found.

This is the role of a Continuous Integration pipeline. The concept is straightforward: every time a developer pushes a branch or opens a pull request, a server automatically checks out the code, builds it, runs the tests, runs the static analysis tools, and reports the results. If any step fails, the change cannot be merged until it is fixed.

A minimal CI configuration for a robotics project using GitHub Actions looks like this:

```yaml
# .github/workflows/static_analysis.yml
name: Static Analysis

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  analyze:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Install tools
        run: |
          sudo apt-get update
          sudo apt-get install -y cppcheck clang-tidy bear gcc-arm-none-eabi

      - name: Build with warnings as errors
        run: |
          make -C build/ \
            CFLAGS="-Wall -Wextra -Wpedantic -Wshadow -Wconversion -Werror"

      - name: Run cppcheck
        run: |
          cppcheck \
            --enable=all \
            --addon=misra \
            --suppressions-list=misra_suppressions.txt \
            --error-exitcode=1 \
            Core/Src/

      - name: Generate compilation database
        run: bear -- make -C build/

      - name: Run clang-tidy
        run: |
          clang-tidy \
            -p compile_commands.json \
            --warnings-as-errors='*' \
            Core/Src/*.c \
            Core/Src/*.cpp
```

Each step in this pipeline corresponds to a layer of the defense in depth strategy. The build step catches compiler-level issues. The cppcheck step enforces MISRA rules. The clang-tidy step enforces modern C++ best practices and naming conventions. A violation at any layer fails the pipeline and prevents the change from being merged.

The suppressions file and the `.clang-tidy` configuration file should both be committed to the repository. This ensures that every developer running the tools locally gets the same results as the CI server, and that the set of active checks is itself under version control and subject to code review. A change that disables a check should go through the same review process as a change that modifies production code.

### STM32CubeIDE: Running cppcheck on the Fault Diagnostics Project

With the fault diagnostics project from Chapter 4 open in CubeIDE, enabling cppcheck analysis takes only a few steps. Right-click the project in the Project Explorer and select Properties. Navigate to C/C++ General → Code Analysis. Check the box labeled Run as you type and also Run on full build. Under Checkers Configuration, find the Cppcheck entry and click Configure.

In the Cppcheck configuration dialog, enable the MISRA C 2012 addon if it appears in your CubeIDE version. If it does not -- CubeIDE's bundled cppcheck version varies -- the command-line approach described earlier is the reliable alternative. The, build the project. Any cppcheck findings appear in the Problems tab alongside compiler warnings, with the rule number and a brief description. Click a finding to jump to the relevant line in the editor.

For a more thorough analysis, the command-line invocation against the project's source directory gives finer control over which checks run and produces output that can be committed to a report file. From a terminal with the project directory as the working directory:

```bash
cppcheck \
  --enable=all \
  --addon=misra \
  --error-exitcode=1 \
  -I Core/Inc \
  --suppress=missingIncludeSystem \
  Core/Src/ \
  2>&1 | tee analysis_report.txtv
```

The `--suppress=missingIncludeSystem` flag suppresses warnings about system headers that cppcheck cannot find, a common source of noise when analyzing embedded projects where standard library headers are in non-standard locations. The `tee` command writes the output both to the terminal and to a report file, which is useful for tracking analysis results over time.

Running this against the fault_handler.c from Chapter 4 will produce several findings. Some will be genuine improvements worth making. Others will be candidates for documented suppressions -- for example, the direct register access patterns required to read the Cortex-M4 fault status registers use pointer casts that MISRA C:2012 Rule 11.3 would flag, and these are legitimate deviations given the hardware access requirements. Working through each finding and deciding whether to fix it or document a suppression is a realistic approximation of the workflow on a professional team.

### Reading a Static Analysis Report

A skill that takes time to develop but matters enormously in practice is reading static analysis output efficiently. A large codebase run through a tool for the first time will often produce hundreds of findings. The temptation is to either fix everything indiscriminately or feel overwhelmed and suppress everything. Neither is right. The productive approach is triage. Mandatory MISRA violations and findings from the `bugprone` category come first. These are most likely to represent real bugs. Required rule violations come next. Advisory violations and style findings like naming conventions come last and may be deferred to a dedicated cleanup pass.

For each finding, the question to ask is: is this a real bug, a latent bug, or a false positive? A real bug needs to be fixed immediately. A latent bug -- code that is technically wrong but happens to work in the current environment -- needs to be fixed before it causes a problem in a different environment. A false positive, where the tool has flagged code that is actually correct, needs to be suppressed with a comment explaining why the suppression is justified. The ratio of real bugs to false positives in a well-configured static analysis run is typically high enough to make the tool valuable even accounting for the time spent reviewing findings. The value increases over time as suppressions accumulate for known false positive patterns and the remaining findings become increasingly actionable.
