Testing is where the design decisions made in every previous chapter either pay off or expose their weaknesses. A codebase with clean component boundaries, a well-defined HAL, explicit preconditions, and documented interfaces is a codebase that is straightforward to test. A codebase where hardware access is scattered throughout the business logic, where functions have implicit global dependencies, and where preconditions are unstated is a codebase where testing is painful, incomplete, and frequently abandoned under schedule pressure.

This chapter covers the testing strategies that matter most for embedded robotics software -- unit testing, hardware-in-the-loop testing, and coverage analysis -- and shows how the structural decisions from the previous chapters make each of them practical rather than aspirational.

### Why Testing Embedded Code Is Different

Testing application software running on a general-purpose computer is relatively straightforward. You run the code on the same machine you develop on, the test framework has access to all the same resources as the production code, and failures produce readable output. Embedded software introduces complications that require deliberate solutions.

- The first complication is the hardware dependency. Production embedded code talks to sensors, actuators, timers, and communication peripherals that do not exist on a development machine. A naive approach to testing requires the target hardware to be physically present, connected, and functional for every test run. This makes tests slow, expensive to run in CI, and fragile -- a hardware fault or a misconfigured peripheral can cause an entire test suite to fail for reasons unrelated to the code being tested.

- The second complication is the execution environment. The target hardware may have limited RAM, no operating system, no standard output, and no filesystem. Test frameworks designed for desktop software assume all of these things are available. Running Google Test on a bare-metal STM32 requires careful configuration and a mechanism for reporting results.

- The third complication is timing. Embedded control systems have real-time constraints. A test that verifies functional correctness but ignores timing behavior may miss an entire class of failures. A PID controller that produces the correct output but takes twice as long as its timing budget is broken in a way that no functional test will catch.

The HAL abstraction from Chapter 7 addresses the first complication directly. The simulation implementations of each HAL interface allow the driver and control layers to be tested on a development machine without hardware. The second complication is addressed by separating tests into two categories: those that run on the host machine against the simulation HAL, and those that run on the target hardware for timing validation and integration testing. The third complication requires measuring execution time on the actual hardware and comparing it against the timing budget -- something that cannot be done in simulation.

### Unit Testing with Google Test

Google Test is the most widely used C++ unit testing framework in the robotics industry. ROS 2 uses it as its standard testing framework. It is well-documented, actively maintained, and integrates cleanly with CMake. The official repository and documentation are at [https://github.com/google/googletest](https://github.com/google/googletest).

Google Test organizes tests into test cases, which are groups of related tests, and individual test functions within each case. Each test function contains one or more assertions -- statements about what a value should be, what condition should be true, or what exception should be thrown.

The assertion macros come in two flavors. `EXPECT_*` macros report a failure but allow the test to continue executing, which is useful when you want to collect multiple failures in a single test run. `ASSERT_*` macros report a failure and immediately abort the current test function, which is appropriate when a failure in one assertion makes subsequent assertions meaningless.

Adding Google Test to the CMake project from Chapter 7 is straightforward using CMake's `FetchContent` module, which downloads and builds the dependency automatically:

```cmake
# tests/CMakeLists.txt
include(FetchContent)

FetchContent_Declare(
    googletest
    GIT_REPOSITORY https://github.com/google/googletest.git
    GIT_TAG        v1.14.0
)

# Prevent GoogleTest from overriding compiler/linker settings
set(gtest_force_shared_crt ON CACHE BOOL "" FORCE)
FetchContent_MakeAvailable(googletest)

enable_testing()

add_executable(robot_arm_tests
    test_pid_controller.cpp
    test_encoder_driver.cpp
    test_kinematics.cpp
)

target_link_libraries(robot_arm_tests
    PRIVATE
    control
    drivers
    kinematics
    hal           # Links against the simulation HAL on host builds
    GTest::gtest_main
)

include(GoogleTest)
gtest_discover_tests(robot_arm_tests)
```

The `gtest_discover_tests` call integrates with CMake's test runner, `ctest`, so that running `ctest` from the build directory executes all registered tests and reports results in a standardized format suitable for CI consumption.

#### Testing the PID Controller

A well-designed unit test for the PID controller from Chapter 8 verifies the documented behavior (the contract stated in the Doxygen comments), rather than the implementation details. This is an important distinction. Tests that verify implementation details break whenever the implementation changes, even if the behavior is unchanged. Tests that verify the contract break only when the contract is violated, which is exactly when you want them to break.

The contract for `PidController::update()` states that the output is clamped to `[-output_limit, output_limit]`, that a positive error produces a positive output for positive gains, and that the integral accumulates over time. Each of these becomes a test:

```cpp
// tests/test_pid_controller.cpp
#include <gtest/gtest.h>
#include "control/pid_controller.hpp"

// Test fixture: shared setup for PID controller tests
class PidControllerTest : public ::testing::Test {
protected:
    void SetUp() override {
        // Construct a controller with known gains for predictable results
        controller_ = std::make_unique<control::PidController>(
            1.0f,    // kp
            0.1f,    // ki
            0.01f,   // kd
            10.0f    // output_limit
        );
    }

    std::unique_ptr<control::PidController> controller_;

    static constexpr float DT_S       = 0.001f;   // 1ms timestep
    static constexpr float TOLERANCE  = 1e-4f;    // Floating point tolerance
};

TEST_F(PidControllerTest, ProportionalOutputMatchesGainTimesError) {
    // With ki=0.1 and kd=0.01, the dominant term at t=0 is proportional.
    // With kp=1.0 and error=5.0, P term = 5.0.
    // I term after one step = 0.1 * 5.0 * 0.001 = 0.0005 (negligible).
    // D term at t=0: prev_error=0, so derivative = 5.0/0.001 = 5000 * 0.01 = 50.
    // This test uses kd=0 to isolate the proportional response.
    control::PidController p_only(1.0f, 0.0f, 0.0f, 100.0f);
    float output = p_only.update(5.0f, 0.0f, DT_S);
    EXPECT_NEAR(output, 5.0f, TOLERANCE);
}

TEST_F(PidControllerTest, OutputIsClamped_WhenErrorExceedsLimit) {
    // A very large error should produce output exactly at the limit,
    // not beyond it. This verifies the saturation postcondition.
    float output = controller_->update(1000.0f, 0.0f, DT_S);
    EXPECT_LE(output,  10.0f);
    EXPECT_GE(output, -10.0f);
}

TEST_F(PidControllerTest, IntegratorAccumulates_OverMultipleSteps) {
    // With a constant error and ki > 0, the integral term should
    // grow over time, increasing the output beyond what the
    // proportional term alone would produce.
    control::PidController pi_only(0.0f, 1.0f, 0.0f, 100.0f);

    float first_output  = pi_only.update(1.0f, 0.0f, DT_S);
    float second_output = pi_only.update(1.0f, 0.0f, DT_S);
    float third_output  = pi_only.update(1.0f, 0.0f, DT_S);

    // Each call accumulates more integral: outputs should be increasing
    EXPECT_LT(first_output,  second_output);
    EXPECT_LT(second_output, third_output);
}

TEST_F(PidControllerTest, ResetIntegrator_ClearsAccumulatedState) {
    // Run the controller for several steps to accumulate integral state
    for (int i = 0; i < 100; ++i) {
        controller_->update(1.0f, 0.0f, DT_S);
    }

    float integral_before_reset = controller_->get_integral();
    EXPECT_GT(std::abs(integral_before_reset), 0.0f);

    controller_->reset_integrator();

    // After reset, a single update with zero error should produce near-zero output
    float output_after_reset = controller_->update(0.0f, 0.0f, DT_S);
    EXPECT_NEAR(output_after_reset, 0.0f, TOLERANCE);
    EXPECT_NEAR(controller_->get_integral(), 0.0f, TOLERANCE);
}

TEST_F(PidControllerTest, NegativeError_ProducesNegativeOutput) {
    // Verify sign convention: negative error produces negative control output
    control::PidController p_only(1.0f, 0.0f, 0.0f, 100.0f);
    float output = p_only.update(-5.0f, 0.0f, DT_S);
    EXPECT_LT(output, 0.0f);
}
```

Notice that each test is focused on a single aspect of behavior, named clearly enough that the test output is self-documenting, and written with explicit comments explaining the reasoning. When a test fails six months from now, the comments tell the engineer diagnosing it exactly what behavior was expected and why.

#### Testing the Encoder Driver with the Simulation HAL

The encoder driver depends on the HAL through the `IUart` interface established in Chapter 7. Testing it uses the `UartSim` simulation implementation, which allows the test to inject controlled data and inspect what the driver transmitted without any hardware:

```cpp
// tests/test_encoder_driver.cpp
#include <gtest/gtest.h>
#include "drivers/encoder_driver.hpp"
#include "hal/uart_sim.hpp"   // Simulation HAL implementation

class EncoderDriverTest : public ::testing::Test {
protected:
    void SetUp() override {
        uart_sim_   = std::make_unique<hal::UartSim>();
        encoder_    = std::make_unique<drivers::EncoderDriver>(*uart_sim_);
    }

    std::unique_ptr<hal::UartSim>           uart_sim_;
    std::unique_ptr<drivers::EncoderDriver> encoder_;
};

TEST_F(EncoderDriverTest, ReadPosition_ReturnsValue_WhenValidDataInjected) {
    // Inject a valid encoder response packet: 4 bytes, little-endian int32
    // representing position 1024
    const uint8_t response[] = { 0x00, 0x04, 0x00, 0x00 };
    uart_sim_->inject_rx_data(response, sizeof(response));

    auto position = encoder_->read_position(0);

    ASSERT_TRUE(position.has_value());
    EXPECT_EQ(position.value(), 1024);
}

TEST_F(EncoderDriverTest, ReadPosition_ReturnsNullopt_ForInvalidJointId) {
    auto position = encoder_->read_position(255);   // Out of range
    EXPECT_FALSE(position.has_value());
}

TEST_F(EncoderDriverTest, ReadPosition_TransmitsCorrectQueryPacket) {
    // Verify that the driver sends the correct query format to the hardware
    const uint8_t response[] = { 0x00, 0x00, 0x00, 0x00 };
    uart_sim_->inject_rx_data(response, sizeof(response));

    encoder_->read_position(2);

    // The query packet should begin with the command byte 0xA1
    // followed by the joint ID
    const auto& tx = uart_sim_->get_tx_buffer();
    ASSERT_GE(tx.size(), 2u);
    EXPECT_EQ(tx[0], 0xA1u);   // Command byte for position query
    EXPECT_EQ(tx[1], 0x02u);   // Joint ID
}
```

These tests run entirely on the development machine. No STM32 board is required. The `UartSim` class absorbs the hardware dependency and makes the driver's behavior fully observable and controllable from the test. This is the direct payoff of the HAL abstraction investment from Chapter 7.

### Parameterized Tests for Boundary Conditions

Boundary conditions (the edges of valid input ranges) are where bugs concentrate. The most important values to test for any numeric parameter are zero, the minimum valid value, the maximum valid value, one below the minimum, and one above the maximum. Writing a separate test function for each of these produces a lot of repetitive code. Google Test's parameterized test facility handles this cleanly:

```cpp
// Parameterized test for PID output clamping at various error magnitudes
struct ClampTestParams {
    float error;
    float expected_sign;   // +1.0f or -1.0f
    bool  should_be_clamped;
};

class PidClampTest
    : public ::testing::TestWithParam<ClampTestParams> {};

TEST_P(PidClampTest, OutputRespectsSaturationLimit) {
    control::PidController controller(10.0f, 0.0f, 0.0f, 5.0f);
    auto params = GetParam();

    float output = controller.update(params.error, 0.0f, 0.001f);

    if (params.should_be_clamped) {
        EXPECT_NEAR(std::abs(output), 5.0f, 1e-4f)
            << "Expected clamped output for error=" << params.error;
    } else {
        EXPECT_LT(std::abs(output), 5.0f)
            << "Expected unclamped output for error=" << params.error;
    }

    if (params.error != 0.0f) {
        EXPECT_EQ(std::signbit(output), std::signbit(params.error))
            << "Output sign should match error sign";
    }
}

INSTANTIATE_TEST_SUITE_P(
    SaturationBoundary,
    PidClampTest,
    ::testing::Values(
        ClampTestParams{ 0.0f,    1.0f,  false },   // Zero error
        ClampTestParams{ 0.1f,    1.0f,  false },   // Small positive error
        ClampTestParams{ 0.5f,    1.0f,  false },   // At limit boundary
        ClampTestParams{ 1.0f,    1.0f,  true  },   // Above limit
        ClampTestParams{ 100.0f,  1.0f,  true  },   // Far above limit
        ClampTestParams{ -0.5f,  -1.0f,  false },   // Negative at boundary
        ClampTestParams{ -1.0f,  -1.0f,  true  }    // Negative above limit
    )
);
```

Each row in the `Values(...)` call produces a separate test case. The test output names each case with its parameters, making failures immediately readable: `SaturationBoundary/PidClampTest.OutputRespectsSaturationLimit/3` corresponds to the fourth parameter set, which has error=1.0f and should be clamped.

### Hardware-in-the-Loop Testing

Unit tests with simulation HAL implementations verify logical correctness. They cannot verify that the STM32 HAL driver configuration is correct, that the encoder hardware returns data in the expected format, that the UART timing is correct at 115200 baud, or that the control loop executes within its timing budget on the actual processor. These properties require testing on the actual hardware, which is the domain of hardware-in-the-loop testing.

Hardware-in-the-loop testing, commonly abbreviated HIL, refers to any test that runs on or against the actual target hardware. It exists on a spectrum from simple on-target unit tests --running the same Google Test suite on the STM32 -- to full system tests where the robot is running and test stimuli are injected through its actual communication interfaces.

#### On-Target Unit Testing with STM32CubeIDE

Running Google Test on a bare-metal STM32 requires redirecting test output to a communication interface and providing a minimal runtime environment. The most practical approach for the STM32F4 Discovery board uses USART2 — already configured in the fault diagnostics project from Chapter 4 -- as the output channel, and implements a minimal `printf` redirect through the STM32 HAL.

First, add a `syscalls` implementation that redirects `write()` to USART2, which allows `printf` and Google Test's output functions to work:

```c
// Core/Src/syscalls.c
#include <errno.h>
#include <sys/unistd.h>
#include "usart.h"

int _write(int file, char* data, int len) {
    if (file == STDOUT_FILENO || file == STDERR_FILENO) {
        HAL_UART_Transmit(&huart2,
                          (uint8_t*)data,
                          (uint16_t)len,
                          HAL_MAX_DELAY);
        return len;
    }
    errno = EBADF;
    return -1;
}
```

Google Test requires dynamic memory allocation and exceptions by default, both of which are problematic on bare metal. The practical solution is to use a minimal test runner that does not depend on these features, or to use the `etl::google_test` compatibility layer from the Embedded Template Library. For a simpler approach that works well for initial HIL testing, a lightweight alternative is to write a minimal test runner manually:

```cpp
// Core/Src/on_target_tests.cpp
#include <cstdio>
#include "control/pid_controller.hpp"

struct TestResult {
    const char* name;
    bool        passed;
};

static uint32_t tests_run    = 0;
static uint32_t tests_passed = 0;

#define ASSERT_NEAR_TEST(name, actual, expected, tol)           \
    do {                                                         \
        ++tests_run;                                             \
        float diff = (actual) - (expected);                     \
        if (diff < 0.0f) diff = -diff;                          \
        bool ok = diff < (tol);                                  \
        if (ok) ++tests_passed;                                  \
        printf("[%s] %s\r\n", ok ? "PASS" : "FAIL", (name));   \
    } while (0)

void run_on_target_tests(void) {
    printf("\r\n=== On-Target Tests ===\r\n");

    /* Test: PID proportional output */
    {
        control::PidController p(1.0f, 0.0f, 0.0f, 100.0f);
        float out = p.update(5.0f, 0.0f, 0.001f);
        ASSERT_NEAR_TEST("PID_proportional_output", out, 5.0f, 1e-3f);
    }

    /* Test: PID output clamping */
    {
        control::PidController p(10.0f, 0.0f, 0.0f, 5.0f);
        float out = p.update(100.0f, 0.0f, 0.001f);
        ASSERT_NEAR_TEST("PID_output_clamped", out, 5.0f, 1e-3f);
    }

    printf("\r\n%lu/%lu tests passed\r\n", tests_passed, tests_run);
    printf("======================\r\n");
}
```

Call `run_on_target_tests()` from `main()` before the main application loop. Flash the firmware and open a serial terminal -- the test results appear before the application starts. This gives immediate confirmation that the logic that passed host-machine unit tests also behaves correctly on the target processor, including any floating point behavior differences between x86 and the Cortex-M4's hardware FPU.

#### Timing Measurement on the STM32F4

Verifying timing behavior requires measuring actual execution time on the hardware. The STM32F4's DWT (Data Watchpoint and Trace) unit provides a free-running cycle counter that can be used for this purpose without consuming a hardware timer:

```c
// Core/Inc/timing.h
#pragma once
#include <stdint.h>
#include "stm32f4xx.h"

/**
 * @brief Initialize the DWT cycle counter.
 * Must be called once before using timing_get_cycles().
 */
static inline void timing_init(void) {
    CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
    DWT->CYCCNT       = 0U;
    DWT->CTRL        |= DWT_CTRL_CYCCNTENA_Msk;
}

/**
 * @brief Return the current DWT cycle count.
 */
static inline uint32_t timing_get_cycles(void) {
    return DWT->CYCCNT;
}

/**
 * @brief Convert a cycle count to microseconds at 168 MHz.
 */
static inline float timing_cycles_to_us(uint32_t cycles) {
    return (float)cycles / 168.0f;
}
```

Using this in a timing test:

```c
void measure_pid_execution_time(void) {
    control::PidController controller(1.0f, 0.1f, 0.01f, 10.0f);

    timing_init();

    uint32_t start  = timing_get_cycles();
    float    output = controller.update(1.0f, 0.0f, 0.001f);
    uint32_t end    = timing_get_cycles();

    uint32_t cycles = end - start;
    float    us     = timing_cycles_to_us(cycles);

    printf("PID update: %lu cycles (%.2f us)\r\n", cycles, (double)us);
    (void)output;

    /* Verify against timing budget: 50us budget for control computation */
    if (us > 50.0f) {
        printf("WARNING: PID update exceeds 50us timing budget\r\n");
    }
}
```

Running this on the STM32F407 at 168 MHz with the FPU enabled gives a concrete cycle count that can be recorded in the module documentation, as discussed in Chapter 8. It also serves as a regression test -- if a future change to the PID implementation increases the execution time past the budget, the warning appears immediately on the next HIL test run.

### Test Coverage Analysis

Testing produces confidence proportional to how much of the code the tests actually exercise. A test suite that only tests the happy path (valid inputs, expected outputs) may achieve high line coverage while leaving entire branches untested. Coverage analysis tools measure how much of the code was executed during a test run and identify which parts were not.

For host-machine tests compiled with GCC, coverage instrumentation is added with two compiler flags:

```cmake
# In tests/CMakeLists.txt, for coverage builds
target_compile_options(robot_arm_tests PRIVATE --coverage -O0)
target_link_options(robot_arm_tests    PRIVATE --coverage)
```

After running the tests, `gcov` processes the instrumentation data and `lcov` produces an HTML coverage report:

```bash
# Run tests
./build/host/tests/robot_arm_tests

# Generate coverage data
lcov --capture \
     --directory build/host/ \
     --output-file coverage.info

# Exclude standard library headers from the report
lcov --remove coverage.info '/usr/*' \
     --output-file coverage_filtered.info

# Generate HTML report
genhtml coverage_filtered.info \
        --output-directory coverage_report/
```

Opening `coverage_report/index.html` shows line-by-line coverage for every source file, with uncovered lines highlighted in red. This report drives the next iteration of test writing -- find the red lines, understand why they are not covered, and write tests that exercise them.

#### MC/DC Coverage for Safety-Critical Code

Line and branch coverage measure whether code was executed. For safety-critical functions at ASIL C and D levels, ISO 26262 requires a stronger criterion called Modified Condition/Decision Coverage, or MC/DC. MC/DC requires that every condition in a boolean expression independently affects the outcome of the decision containing it.

The distinction matters for expressions with multiple conditions. Consider the output clamping logic in the PID controller:

```cpp
if (output > output_limit_ || output < -output_limit_) {
    output = std::clamp(output, -output_limit_, output_limit_);
}
```

Branch coverage requires that the `if` condition is both true and false at least once. MC/DC requires additionally that there exist test cases where:

- `output > output_limit_` is true and `output < -output_limit_` is false, causing the overall condition to be true
- `output > output_limit_` is false and `output < -output_limit_` is true, causing the overall condition to be true
- Both conditions are false, causing the overall condition to be false

Each condition must be shown to independently affect the outcome. The parameterized clamping tests written earlier in this chapter already satisfy MC/DC for this expression, because the parameter set includes positive clamping, negative clamping, and no clamping cases independently. Designing tests with MC/DC in mind from the start, rather than retrofitting coverage after the fact, produces a tighter and more purposeful test suite.

### Structuring Tests for Dual Deployment

The testing architecture that emerges from this chapter has two deployment targets: host-machine tests for rapid development feedback, and on-target tests for hardware validation and timing verification. Managing these in the same CMake project requires a small amount of structure.

The key insight is that the test logic itself -- the assertions, the parameterized inputs, the expected outputs -- should be identical in both deployments. What differs is the test runner, the HAL implementation, and the output mechanism. A clean way to achieve this is to separate test logic from test infrastructure:

```
tests/
├── CMakeLists.txt
├── host/                    # Host-machine test runner (Google Test)
│   ├── CMakeLists.txt
│   └── main_host.cpp
├── target/                  # On-target test runner (lightweight)
│   ├── CMakeLists.txt
│   └── main_target.cpp
└── cases/                   # Shared test logic (no runner dependency)
    ├── test_pid_controller.cpp
    ├── test_encoder_driver.cpp
    └── test_kinematics.cpp
```

The `cases/` directory contains the test logic expressed in a way that both runners can compile. The host runner links against Google Test and provides `main()` through `GTest::gtest_main`. The target runner provides its own `main()` that calls into the same test functions through a lightweight registration mechanism.

This structure means that every test written for the host machine is also available for on-target validation. If a test passes on the host and fails on the target, the discrepancy points directly at a hardware-specific behavior that the simulation HAL did not model correctly -- which is exactly the kind of information that makes HIL testing valuable.

### Integrating Testing into CI

The testing pipeline in CI extends the static analysis pipeline from Chapter 6 with two additional stages. The host-machine unit tests run on every push, providing rapid feedback. A coverage report is generated and published, with a minimum coverage threshold that fails the build if coverage drops below an acceptable level.

```yaml
# Addition to .github/workflows/static_analysis.yml

  test:
    runs-on: ubuntu-latest
    needs: analyze

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Install dependencies
        run: |
          sudo apt-get update
          sudo apt-get install -y cmake gcc lcov

      - name: Build host tests with coverage
        run: |
          cmake -B build/host \
                -DCMAKE_BUILD_TYPE=Debug \
                -DENABLE_COVERAGE=ON
          cmake --build build/host

      - name: Run unit tests
        run: |
          cd build/host
          ctest --output-on-failure

      - name: Generate coverage report
        run: |
          lcov --capture \
               --directory build/host/ \
               --output-file coverage.info
          lcov --remove coverage.info '/usr/*' \
               --output-file coverage_filtered.info
          genhtml coverage_filtered.info \
                  --output-directory coverage_report/

      - name: Check coverage threshold
        run: |
          COVERAGE=$(lcov --summary coverage_filtered.info 2>&1 \
                     | grep "lines" \
                     | grep -o '[0-9.]*%' \
                     | head -1 \
                     | tr -d '%')
          echo "Line coverage: ${COVERAGE}%"
          if (( $(echo "$COVERAGE < 80" | bc -l) )); then
            echo "Coverage ${COVERAGE}% is below 80% threshold"
            exit 1
          fi

      - name: Upload coverage report
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage_report/
```

The 80% threshold in the coverage check is a starting point, not a universal standard. Safety-critical functions at ASIL C and D may require 100% MC/DC coverage. Less critical utility code may be adequately served by 70% line coverage. The threshold should reflect the safety requirements of the codebase and should be reviewed whenever a new module is added.
