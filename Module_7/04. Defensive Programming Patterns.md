The previous two chapters established why C and C++ can be  dangerous languages for production systems and what the industry's response to that danger looks like in the form of coding standards. This chapter is where that knowledge becomes a practical toolkit. Defensive programming is a methodology expressed through a collection of patterns that you apply consistently throughout a codebase, so that when something unexpected happens, your software fails informatively.

The patterns covered here are not exotic. Several of them will look familiar. What changes at the professional level is not the individual techniques but the discipline with which they are applied -- every function, every interface boundary, every piece of data that arrives from the outside world.

### Preconditions, Postconditions, and the Contract Model

Every function makes an implicit deal with its callers. The caller agrees to provide inputs within a certain range and in a certain state. The function agrees to produce a correct result when those conditions are met. In amateur code this deal is informal and undocumented. In production code it is explicit, and enforcing it is the function's responsibility.

The formal terms for this are preconditions and postconditions, borrowed from Bertrand Meyer's design-by-contract methodology. A precondition is something that must be true when a function is called for the function's behavior to be defined. A postcondition is something the function guarantees will be true when it returns, provided the preconditions were satisfied.

Getting into the habit of stating these explicitly; first in your head, then in comments, then in code, changes how you write functions. Consider a PID controller update function:

```cpp
#include <cstdint>
#include <cassert>

/**
 * @brief Update PID controller and compute new control output.
 *
 * @pre dt_ms > 0
 * @pre setpoint is finite
 * @pre measurement is finite
 * @post return value is in range [-output_limit, output_limit]
 */
float pid_update(float setpoint, float measurement, float dt_ms) {
    assert(dt_ms > 0.0f);
    assert(output_limit > 0.0f);

    float error = setpoint - measurement;
    integral_ += error * dt_ms;
    float derivative = (error - prev_error_) / dt_ms;
    prev_error_ = error;

    float output = (kp_ * error) + (ki_ * integral_) + (kd_ * derivative);

    /* Clamp output to valid range */
    if (output > output_limit_)  output = output_limit_;
    if (output < -output_limit_) output = -output_limit_;

    assert(output >= -output_limit_ && output <= output_limit_);
    return output;
}
```

The `assert` calls at the entry point enforce the preconditions. The `assert` at the exit point verifies the postcondition. The comment block documents the contract for anyone reading the header. None of this is complicated, but together it means that if this function is ever called with a zero or negative timestep, which would produce a division by zero and nonsensical derivative output -- the failure is caught immediately at the point of the violation rather than propagating into the control loop.

Assertions are a development and testing tool. The standard `assert` macro from `<cassert>` is typically compiled out in release builds with `NDEBUG` defined, which is the correct behavior for most embedded targets where you want the overhead eliminated in production firmware. But this raises an important question: **what happens in production when a precondition is violated?**

The answer is that assertions and production error handling serve different purposes and should coexist. Assertions catch programming errors -- bugs where the code is being called incorrectly. Production error handling catches runtime errors -- unexpected inputs, hardware failures, communication errors. A function should use **assertions** to verify things that _should_ never be wrong given correct code, and explicit **error return mechanisms** to handle things that _might_ be wrong at runtime regardless of code correctness.

### Error Handling Strategies

The embedded and robotics C++ world has not converged on a single error handling strategy, and for good reason -- different contexts have genuinely different requirements. Understanding the tradeoffs helps you make the right choice for each part of a codebase.

C-style return codes are the oldest mechanism and remain appropriate in low-level hardware drivers and any code path where deterministic execution time is essential. The pattern is simple: functions return an error code type, callers check it, and error information propagates up the call stack manually. The weakness is that return codes are easy to ignore, and an ignored error code produces no diagnostic at all.

```c
typedef enum {
    STATUS_OK        = 0,
    STATUS_NULL_PTR  = 1,
    STATUS_OUT_OF_RANGE = 2,
    STATUS_TIMEOUT   = 3
} status_t;

status_t read_encoder(uint8_t joint_id, int32_t* position_out) {
    if (position_out == NULL) return STATUS_NULL_PTR;
    if (joint_id >= NUM_JOINTS) return STATUS_OUT_OF_RANGE;

    *position_out = encoder_registers[joint_id];
    return STATUS_OK;
}
```

C++ exceptions are the language's built-in mechanism for propagating errors and are appropriate in application-layer robotics code — path planning, perception pipelines, configuration loading — where deterministic timing is not a hard requirement. They are not appropriate in hard real-time control loops because stack unwinding has nondeterministic cost and because many embedded standard library implementations do not support exceptions reliably. AUTOSAR Rule A15-0-1 reflects this: exceptions shall not be used in safety-critical real-time code.

`std::optional` and `std::expected` occupy a useful middle ground. `std::optional<T>` expresses a value that may or may not be present without requiring heap allocation or exceptions. `std::expected<T, E>`, standardized in C++23, goes further by carrying either a value or an error code in a single return type, making the error handling path part of the function's type signature and impossible to ignore without a deliberate choice.

```cpp
#include <optional>
#include <cstdint>

std::optional<int32_t> read_encoder(uint8_t joint_id) {
    if (joint_id >= NUM_JOINTS) return std::nullopt;
    return encoder_registers[joint_id];
}

/* At the call site, the caller must handle the absent case explicitly */
auto position = read_encoder(joint_id);
if (!position.has_value()) {
    log_error("Invalid joint ID: %u", joint_id);
    return;
}

move_to_position(position.value());
```

The `[[nodiscard]]` attribute, introduced in C++17, is a valuable complement to any of these strategies. Applying it to a function's return type causes the compiler to emit a warning if the caller discards the return value. For functions whose return value encodes success or failure, this is exactly the right behavior; ignoring a status return is almost always a bug.

```cpp
[[nodiscard]] status_t initialize_motor_driver(uint8_t motor_id);

/* This line now produces a compiler warning: return value discarded */
initialize_motor_driver(0);

/* This is correct: caller acknowledges the return value */
if (initialize_motor_driver(0) != STATUS_OK) {
    enter_safe_state();
}
```

Making `[[nodiscard]]` a team convention for all functions that return status codes eliminates an entire class of failure modes with no runtime cost.

### Strong Typedefs and Unit Safety

One of the most instructive engineering disasters in history was caused by a unit mismatch. In 1999, the Mars Climate Orbiter was lost because one engineering team used metric units and another used imperial units in data passed between software systems. The spacecraft entered the Martian atmosphere at the wrong angle and was destroyed.

The underlying problem is that C and C++ numeric types carry no semantic information. A `float` is a `float` whether it represents meters, feet, radians, degrees, newtons, or pounds. The type system offers no protection against passing a value in the wrong unit to a function that expects a different unit.

Strong typedefs solve this by wrapping numeric types in distinct named types that the compiler treats as incompatible even though the underlying representation is the same. In modern C++, the cleanest way to implement this is with a simple templated wrapper:

```cpp
template <typename T, typename Tag>
struct StrongType {
    explicit StrongType(T value) : value_(value) {}
    T get() const { return value_; }
private:
    T value_;
};

/* Define distinct types for each physical quantity */
struct RadiansTag {};
struct DegreesTag {};
struct MetersTag {};
struct MetersPerSecondTag {};

using Radians          = StrongType<float, RadiansTag>;
using Degrees          = StrongType<float, DegreesTag>;
using Meters           = StrongType<float, MetersTag>;
using MetersPerSecond  = StrongType<float, MetersPerSecondTag>;
```

Now functions that operate on angles can declare their parameters in the correct unit type, and the compiler rejects calls that pass the wrong unit:

```cpp
void set_joint_angle(uint8_t joint_id, Radians angle);

Degrees user_input(45.0f);
Radians converted(user_input.get() * M_PI / 180.0f);

set_joint_angle(0, converted);    /* Correct */
set_joint_angle(0, user_input);   /* Compiler error: wrong type */
```

The compiler error on the last line is the feature. The bug that would have commanded a joint to move to 45 radians instead of 45 degrees -- which on most robotic arms would immediately trigger a fault or cause physical damage -- is caught at compile time with zero runtime cost.

This pattern adds some verbosity, but in a robotics codebase that moves physical actuators based on computed values, the cost of that verbosity is trivially small compared to the cost of a unit error in production.

### Safe Numeric Patterns for Embedded Robotics

Sensor data in robotics systems is rarely clean. Values arrive out of range due to hardware faults, communication errors, or simply because the physical situation exceeds the sensor's measurement range. Production code should treat every piece of incoming numeric data as potentially invalid until it has been explicitly validated.

Clamping is the most common form of numeric safety for actuator commands. Before any computed value is written to a hardware interface, it should be constrained to the physically and electrically safe range of that interface:

```cpp
#include <algorithm>
#include <cstdint>

constexpr float VELOCITY_MIN_MPS = -2.0f;
constexpr float VELOCITY_MAX_MPS =  2.0f;

float sanitize_velocity_command(float commanded_velocity) {
    return std::clamp(commanded_velocity, VELOCITY_MIN_MPS, VELOCITY_MAX_MPS);
}
```

Clamping is not a substitute for understanding where an out-of-range value came from, but it is a last line of defense. If your controller is regularly producing commands that require clamping, something upstream is wrong and should be investigated. A good practice is to log or count clamping events so they are visible in diagnostics:

```cpp
float sanitize_velocity_command(float commanded_velocity) {
    float clamped = std::clamp(commanded_velocity, VELOCITY_MIN_MPS, VELOCITY_MAX_MPS);

    if (clamped != commanded_velocity) {
        diagnostics_.velocity_clamp_events++;
        log_warning("Velocity command %.3f clamped to %.3f",
                    commanded_velocity, clamped);
    }

    return clamped;
}
```

For integer arithmetic in control code, the saturating addition utility from Chapter 3 is the right pattern. For floating point code, it is worth explicitly checking for NaN and infinity before using a value in any computation. A single NaN propagates through an entire calculation chain, producing NaN outputs that may be clamped to a limit value and sent to an actuator without anyone noticing the upstream corruption:

```cpp
#include <cmath>

bool is_valid_sensor_reading(float value) {
    return std::isfinite(value);   /* Returns false for NaN and infinity */
}

float process_imu_acceleration(float raw_value) {
    if (!is_valid_sensor_reading(raw_value)) {
        log_error("Invalid IMU reading: %f", raw_value);
        return 0.0f;   /* Safe fallback */
    }
    return raw_value * IMU_SCALE_FACTOR;
}
```

### STM32CubeIDE Tutorial: Observing and Diagnosing Hardware Faults

Everything discussed so far in this chapter is about preventing failures at the software level. But on an embedded target, when something does go wrong: a null pointer dereference, a buffer overrun past the end of the stack, a division by zero, the hardware itself responds. On the Cortex-M4 inside your STM32F407, this response takes the form of a fault exception. Understanding what the hardware tells you when it faults, and how to read that information, is an essential skill for embedded robotics development.

This tutorial walks through setting up a diagnostic HardFault handler on the STM32F4 Discovery board, deliberately triggering two of the fault conditions described in Chapter 3, and using the fault status registers to identify exactly what went wrong.

#### Setting Up the Project

Open STM32CubeIDE and create a new STM32 project. When the target selector appears, search for STM32F407VGTx and select it. Name the project something like `fault_diagnostics`. Accept the default peripheral initialization -- for this tutorial you only need the core and the UART for debug output. In the `.ioc` configurator, enable USART2 in asynchronous mode at 115200 baud. 

Generate the code and open the project. You will work primarily in `Core/Src/main.c` and a new file you will create called `fault_handler.c`.

#### Understanding the Fault Status Registers

When the Cortex-M4 encounters a fault, it stores diagnostic information in a set of registers before jumping to the fault handler. These registers tell you precisely what kind of fault occurred and, in many cases, where in memory it occurred. The four registers you will use are:

- **The Configurable Fault Status Register (CFSR)** at address `0xE000ED28` is a composite register containing three sub-registers. The MemManage Fault Status bits indicate memory protection unit violations. The BusFault Status bits indicate bus errors such as accesses to invalid memory addresses. The UsageFault Status bits indicate instruction-level faults such as division by zero and unaligned access.

- **The HardFault Status Register (HFSR)** at address `0xE000ED2C` indicates whether the hard fault was escalated from a configurable fault, which happens when a configurable fault occurs but its handler is not enabled.

- **The MemManage Fault Address Register (MMFAR)** at address `0xE000ED34` contains the address that caused a memory management fault, when the MMARVALID bit in CFSR is set.

- **The BusFault Address Register (BFAR)** at address `0xE000ED38` contains the address that caused a bus fault, when the BFARVALID bit in CFSR is set.

Create `fault_handler.c` in `Core/Src/` and add the following:

```c
#include <stdint.h>
#include <stdio.h>
#include <string.h>
#include "stm32f4xx_hal.h"

/* huart2 is declared in main.c by CubeMX; expose it here */
extern UART_HandleTypeDef huart2;

/* Fault status register addresses */
#define CFSR   (*((volatile uint32_t*)0xE000ED28))
#define HFSR   (*((volatile uint32_t*)0xE000ED2C))
#define MMFAR  (*((volatile uint32_t*)0xE000ED34))
#define BFAR   (*((volatile uint32_t*)0xE000ED38))

/* CFSR UsageFault bits */
#define CFSR_DIVBYZERO  (1U << 25)
#define CFSR_UNALIGNED  (1U << 24)
#define CFSR_UNDEFINSTR (1U << 16)

/* CFSR BusFault bits */
#define CFSR_BFARVALID  (1U << 15)
#define CFSR_PRECISERR  (1U << 9)
#define CFSR_IBUSERR    (1U << 8)

/* CFSR MemManage bits */
#define CFSR_MMARVALID  (1U << 7)
#define CFSR_DACCVIOL   (1U << 1)

static void fault_print(const char* msg) {
    HAL_UART_Transmit(&huart2,
                      (uint8_t*)msg,
                      strlen(msg),
                      HAL_MAX_DELAY);
}

void HardFault_Handler(void) {
    char buf[128];

    fault_print("\r\n=== HARDFAULT DIAGNOSTIC ===\r\n");

    snprintf(buf, sizeof(buf), "CFSR  : 0x%08lX\r\n", CFSR);
    fault_print(buf);

    snprintf(buf, sizeof(buf), "HFSR  : 0x%08lX\r\n", HFSR);
    fault_print(buf);

    /* Decode UsageFault causes */
    if (CFSR & CFSR_DIVBYZERO) {
        fault_print("CAUSE : Division by zero\r\n");
    }
    if (CFSR & CFSR_UNALIGNED) {
        fault_print("CAUSE : Unaligned memory access\r\n");
    }
    if (CFSR & CFSR_UNDEFINSTR) {
        fault_print("CAUSE : Undefined instruction\r\n");
    }

    /* Decode BusFault causes */
    if (CFSR & CFSR_PRECISERR) {
        snprintf(buf, sizeof(buf), "CAUSE : Precise bus fault at address 0x%08lX\r\n",
                 (CFSR & CFSR_BFARVALID) ? BFAR : 0xFFFFFFFF);
        fault_print(buf);
    }

    /* Decode MemManage causes */
    if (CFSR & CFSR_DACCVIOL) {
        snprintf(buf, sizeof(buf), "CAUSE : Memory access violation at 0x%08lX\r\n",
                 (CFSR & CFSR_MMARVALID) ? MMFAR : 0xFFFFFFFF);
        fault_print(buf);
    }

    fault_print("============================\r\n");

    /* Halt: in production this would trigger a safe state and watchdog reset */
    while (1) {}
}
```


Before this will compile, CubeIDE's generated `stm32f4xx_it.c` contains a default `HardFault_Handler` that consists of an infinite loop. You need to comment it out or delete it, otherwise you will get a multiple definition linker error. Open `Core/Src/stm32f4xx_it.c`, find the `HardFault_Handler` function, and remove its body, leaving only the declaration, or delete it entirely.

You also need to enable the division by zero trap in the Cortex-M4's Configuration and Control Register (CCR). Add this to the start of your `main()` function, before the main loop:

```c
/* Enable divide-by-zero and unaligned access traps */
SCB->CCR |= SCB_CCR_DIV_0_TRP_Msk | SCB_CCR_UNALIGN_TRP_Msk;
```

Without this, integer division by zero on the Cortex-M4 silently returns zero rather than faulting, which is a behavior that would hide exactly the class of bugs you are trying to detect.

#### Triggering a Division by Zero Fault

Add this function to `main.c`, outside the main loop:

```c
static void trigger_div_zero(void) {
    volatile int32_t numerator   = 100;
    volatile int32_t denominator = 0;
    volatile int32_t result      = numerator / denominator;
    (void)result;
}
```

The `volatile` qualifiers are important here. Without them, a compiler at any optimization level is permitted to see that the denominator is a constant zero and either optimize the division away entirely or flag it at compile time. `volatile` forces the compiler to generate an actual division instruction. Call `trigger_div_zero()` from inside `main()` before the while loop, build the project, and flash it to the board.

Open a serial terminal at 115200 baud on the Discovery board's virtual COM port. When the firmware runs, you should see output similar to:

```
=== HARDFAULT DIAGNOSTIC ===
CFSR  : 0x02000000
HFSR  : 0x40000000
CAUSE : Division by zero
============================
```

The CFSR value `0x02000000` has bit 25 set, which corresponds to `DIVBYZERO` in the UsageFault byte. The HFSR value `0x40000000` has bit 30 set, which is `FORCED`, indicating that a configurable fault (the UsageFault) was escalated to a HardFault because the UsageFault handler was not individually enabled. This is normal behavior when you have not configured separate fault handlers.

#### Triggering a Null Pointer Dereference

Remove or comment out the `trigger_div_zero()` call and add this instead:

```c
static void trigger_null_deref(void) {
    volatile uint32_t* null_ptr = NULL;
    volatile uint32_t  value    = *null_ptr;   /* Read from address 0x00000000 */
    (void)value;
}
```

On the STM32F407, address `0x00000000` is not mapped to valid RAM or flash in the default memory configuration. Dereferencing a null pointer therefore causes a bus fault. Flash and run. Your terminal should now show:

```
=== HARDFAULT DIAGNOSTIC ===
CFSR  : 0x00008200
HFSR  : 0x40000000
CAUSE : Precise bus fault at address 0x00000000
============================
```

The CFSR value has bits 9 and 15 set; `PRECISERR` indicating a precise bus fault, and `BFARVALID` indicating that the BFAR register contains the faulting address. The handler correctly reports address `0x00000000`, giving you an immediate, actionable diagnosis.

#### What This Means in Practice

This diagnostic handler is a development and debugging tool. In production firmware, a HardFault handler has different responsibilities. It should save enough system state to diagnose the fault post-mortem -- typically into a dedicated region of non-volatile RAM or flash that survives a reset -- trigger a watchdog-supervised reset to restore operation, and log the fault event to whatever telemetry system the robot uses.

The deeper lesson is that every undefined behavior category from Chapter 3 has a corresponding hardware fault that this handler can detect and identify. Division by zero maps to `DIVBYZERO` in the UsageFault byte. Null pointer dereference and out-of-bounds accesses to unmapped memory map to `PRECISERR` in the BusFault byte. Enabling the Memory Protection Unit (MPU), which the STM32F4 supports and which CubeIDE can configure through the `.ioc` file, allows stack overflow detection to map to `DACCVIOL` in the MemManage byte.

Knowing your hardware tells you when your software has violated the contract established in Chapters 2 and 3 is not just reassuring. It makes the abstract concept of undefined behavior concrete and immediately detectable, which changes how you think about writing the code in the first place.

### Putting It Together

The patterns in this chapter form a system. Strong typedefs prevent unit errors from reaching the hardware. Bounds-checked input validation prevents corrupt sensor data from propagating into the control pipeline. Explicit precondition checking ensures that functions are called correctly. Saturating arithmetic and NaN validation prevent numeric corruption from producing plausible-looking wrong outputs. And the HardFault handler ensures that when all of those defenses are insufficient, the failure is immediately visible and diagnosable rather than silent.

None of these patterns are expensive in terms of code size or execution time when applied thoughtfully. The bounds checks and precondition assertions in a control loop are a handful of compare-and-branch instructions. The strong typedef wrappers have zero runtime representation. The NaN check is a single floating point instruction. The cost of not having them, as every field failure story in this module illustrates, is vastly higher.
