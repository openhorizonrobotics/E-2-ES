Every technique covered in this module: the coding standards, the defensive patterns, the static analysis pipeline, the testing strategies -- operates at the level of individual code changes. Code review is the process that connects those individual changes to the health of the codebase as a whole. It is where accumulated knowledge is transmitted between team members, where mistakes that automated tools cannot catch are found, and where the engineering culture of a team becomes visible in concrete form.

This final chapter covers what professional code review looks like in a safety-critical robotics context, how technical debt accumulates and is managed without stopping feature development, and how the individual practices from every preceding chapter combine into a coherent development workflow. By the end, the goal is not just that you know a collection of techniques -- it is that you understand how they fit together into a system that a team can sustain over years rather than months.

### What Code Review Is Actually For

A common misconception about code review is that its primary purpose is catching bugs. Bug detection is valuable, but it is not the most important thing code review does. Static analysis tools catch bugs more consistently and more cheaply than human reviewers. The things code review does that automated tools cannot are more subtle and more valuable over the long term.

1. **Code review transfers knowledge.** When a reviewer reads a change and asks why a particular design decision was made, the answer -- whether it comes in a comment, in an updated ADR, or in a conversation -- becomes part of the team's shared understanding. A codebase where every significant decision has been reviewed is a codebase where multiple people understand why the code is the way it is. This is not a luxury. It is the primary defense against the situation where one engineer leaves the team and takes irreplaceable knowledge with them.

2. **Code review enforces consistency.** A codebase written by ten engineers without review becomes ten different coding styles, ten different approaches to error handling, ten different levels of documentation completeness. Consistency matters in production software not for aesthetic reasons but for practical ones -- consistent code is faster to read, easier to analyze statically, and less likely to contain the kind of subtle contextual bug that arises when one module makes an assumption that another module violates.

3. **Code review surfaces design problems early.** A bug in a function body is cheap to fix. A design decision that affects the interface between modules is expensive to fix after the fact because it may require changes across many files and many callers. A reviewer who spots a problematic interface design in a pull request saves far more time than a reviewer who spots a missing bounds check.

### The Anatomy of a Good Pull Request

Code review begins before a reviewer reads a single line. The quality of a pull request -- the unit of change submitted for review -- determines how useful the review can be.

A pull request should represent a single coherent change. Not a collection of unrelated fixes bundled together for convenience, not a multi-week feature implemented in a single enormous diff, but one thing: one new feature, one bug fix, one refactoring. This constraint feels limiting until you experience the alternative -- reviewing a 2000-line pull request that changes the HAL interface, fixes an encoder driver bug, and adds a new control mode simultaneously. Each of those changes deserves its own review, its own tests, and its own commit history.

The pull request description is as important as the code. A reviewer who understands why a change was made reviews it more effectively than one who must infer the motivation from the diff alone. A good description answers three questions: what does this change do, why was it necessary, and what was the approach. For changes that involved non-obvious design decisions, a reference to the relevant ADR or a brief inline explanation of alternatives considered is appropriate.

For the robotic arm project from the previous chapters, a well-written pull request description might look like this:

```
## Add anti-windup to PID controller

### What
Adds conditional integration (back-calculation anti-windup) to
PidController to prevent integrator windup during output saturation.

### Why
During prolonged saturation — for example when a joint is held
against a hard stop — the integrator accumulates a large value.
When saturation ends, this produces an overshoot spike before the
integrator unwinds. Field testing on joint 3 showed 0.3 rad
overshoot after release from the end stop.

### Approach
Back-calculation anti-windup: when output is saturated, the
integrator is updated with a corrected error term that accounts
for the difference between the unsaturated and saturated output.
The anti-windup gain (kb) is set to ki/kp as recommended in
Astrom & Hagglund "PID Controllers: Theory, Design and Tuning".

Alternative considered: integrator clamping. Rejected because it
causes integrator reset on every saturation event, which produces
sluggish response when saturation ends. See ADR-007 for the full
comparison.

### Testing
- Added unit tests for anti-windup behavior in test_pid_controller.cpp
- Verified on STM32F407 target: joint 3 overshoot reduced from 0.3
  to 0.04 rad in on-target timing test
- No change to worst-case execution time (measured: 4.3us, budget: 50us)
```

A reviewer reading this before looking at a single line of code already understands the problem, the solution, the alternatives considered, and the evidence that it works. The review can focus on the quality of the implementation rather than reconstructing the context.

### Reviewing Safety-Critical Code

Code review in a safety-critical robotics context has a different character from review in typical software development. The reviewer's responsibility extends beyond correctness to include a judgment about whether the change maintains the safety properties of the system.

This does not mean that every code reviewer needs to be a safety engineer. It means that reviewers in a safety-critical codebase develop a specific set of habits that go beyond checking for bugs.

- The first habit is verifying that preconditions are checked. Every function that receives data from outside its module (sensor readings, communication packets, user commands), should validate that data at the point of entry. A reviewer who sees a function calling `compute_joint_torque(sensor_value)` without a prior validity check should ask: what happens if `sensor_value` is NaN? What if it is out of the physically meaningful range? The answer should be visible in the code, not assumed.

- The second habit is tracing error paths. When a function can fail -- returns a status code, returns `std::nullopt`, throws an exception -- the reviewer should follow every call site and verify that the failure is handled. A common failure pattern is code that checks a return value in the first call site but not in subsequent ones added later. Static analysis catches some of these, but not all.

- The third habit is checking resource cleanup. For any change that acquires a resource -- enables a motor, opens a communication channel, locks a mutex -- the reviewer should verify that the corresponding release is guaranteed under all exit paths. The RAII patterns from Chapter 5 make this mechanical to check: if the acquisition is in a constructor and the release is in a destructor, the analysis reduces to verifying that the object's lifetime covers all the code paths that need it.

- The fourth habit is questioning magic numbers. A numeric constant in a change that is not either named, documented, or traceable to a specification deserves a comment. The reviewer should ask: where does this value come from? Is it a physical constant, a hardware specification, a tuned parameter? If the answer is not obvious, the code needs a comment.

For a safety-critical change -- one that affects an ASIL-classified component -- most organizations require a minimum of two reviewers and mandate that at least one reviewer is independent of the author. Independence means the reviewer was not involved in the design of the change and brings genuinely fresh eyes to the review. ISO 26262 Part 6 uses the term "independence" formally in this context, and at higher ASIL levels it requires that the reviewer and the author work in different organizational units.

### What to Look For: A Reviewer's Checklist

Rather than reviewing code with a general sense of "does this seem okay," experienced reviewers work through a mental checklist. Making this checklist explicit is useful both for reviewers and for authors preparing a change for review -- knowing what reviewers look for helps authors address the most common issues before submitting.

For any change to production robotics code, the checklist covers the following areas.

- **Correctness.** Does the implementation match the stated intent? Are there off-by-one errors, sign errors, or unit mismatches? Are boundary conditions handled -- zero inputs, maximum inputs, empty containers?

- **Contract compliance.** Are preconditions checked? Are postconditions verifiable from the implementation? Does the Doxygen comment accurately describe the function's behavior, including the failure cases?

- **Error handling.** Are all fallible operations checked? Are error codes propagated appropriately? Is there a safe state to enter if an unrecoverable error occurs?

- **Resource management.** Are all acquired resources released? Are RAII patterns used correctly? Are there any paths where a resource could be acquired twice?

- **Coding standard compliance.** Does the change introduce any new MISRA or AUTOSAR violations? Has the static analysis pipeline been run against the change? Are there undocumented deviations?

- **Test coverage.** Does the change include tests for its new behavior? Are boundary conditions tested? Does the test coverage include both the success path and the failure paths?

- **Documentation.** Are new public functions documented with Doxygen? Are non-obvious implementation choices explained with comments? If a significant design decision was made, is there an ADR?

- **Timing impact.** For changes to real-time code paths, has the worst-case execution time been measured? Is it within the timing budget?

Not every item on this checklist applies to every change. A documentation fix does not need a timing analysis. A change to a non-real-time configuration parser does not need MC/DC coverage analysis. The checklist is a reminder of what to consider, not a mandatory form to fill out.

### Technical Debt in Robotics Codebases

Technical debt is the accumulated cost of shortcuts, outdated design decisions, and deferred improvements in a codebase. Every production codebase accumulates it. The question is not whether a robotics codebase will have technical debt but how the team manages it -- whether it accumulates in a controlled, visible way or silently until it causes a crisis.

The metaphor of debt is precise in a way that is worth taking seriously. Like financial debt, technical debt has a principal -- the cost of fixing the underlying issue -- and interest -- the ongoing cost of working around it every day it remains unfixed. A function with unclear preconditions has a small principal to fix but charges interest every time a new caller makes a wrong assumption about it. A module with circular dependencies has a high principal to fix and charges interest every time a change requires touching both sides of the cycle.

In robotics codebases specifically, technical debt often accumulates in predictable places.

- The first is the HAL. Hardware abstraction layers written quickly for an initial prototype often have leaky abstractions -- places where hardware-specific behavior bleeds through the interface into the code that should be portable. The interest on this debt is paid every time the project is ported to new hardware or tested in simulation. 
- The second is the startup and initialization sequence. Initialization code is written once, usually under schedule pressure, and is rarely revisited. It accumulates assumptions about hardware state, initialization ordering, and timing that are not documented and are easy to violate inadvertently. 
- The third is configuration management -- the ad hoc collections of `#define` constants and globally accessible structs that grow organically and become impossible to reason about statically.

#### Tracking Debt Explicitly

The most effective way to prevent technical debt from becoming invisible is to track it explicitly. A technical debt register -- a file in the repository, typically a simple markdown document -- lists known issues with enough context to prioritize and plan the work to address them.

```markdown
# Technical Debt Register

## TD-001: HAL UART interface does not model DMA transfers
**File:** hal/include/hal/uart.hpp
**Introduced:** 2024-02-10
**Interest:** Every high-throughput communication feature
requires platform-specific workarounds outside the HAL.
**Fix:** Extend IUart with async transmit/receive using
callbacks. Estimated effort: 3 days.
**Priority:** Medium — address before adding lidar driver.

## TD-002: PID gains stored as global constants in pid_config.h
**File:** control/include/control/pid_config.h
**Introduced:** 2024-01-05
**Interest:** Cannot run multiple controller instances with
different gains. Makes automated gain tuning impossible.
Cannot be validated by static analysis.
**Fix:** Move gains into PidController constructor parameters
and load from a typed configuration struct.
Estimated effort: 1 day + test updates.
**Priority:** High — blocking automated tuning feature.
```

A debt register like this does several useful things simultaneously. It makes debt visible to the entire team rather than existing only in the memory of whoever introduced it. It captures the interest -- the ongoing cost of leaving the issue unaddressed -- which is essential for prioritization. And it provides the context needed for a new engineer to understand why the code is the way it is rather than assuming it is the correct design.

Addressing technical debt should be treated as engineering work with the same planning and review discipline as feature development. A common antipattern is declaring a "refactoring sprint" with the intent to fix all accumulated debt at once. This rarely works because the scope is too large and the pressure to return to feature development too immediate. A more sustainable approach is to allocate a fixed fraction of each development cycle -- ten to twenty percent -- to debt reduction and treat this allocation as non-negotiable.

### Version Control as an Engineering Discipline

Version control was mentioned briefly in Chapter 7 in the context of project structure. At the level of a team workflow, it deserves more thorough treatment, because the discipline with which a team uses version control determines how well the codebase can be audited, debugged, and maintained.

Commit history is a form of documentation. A well-maintained commit history is a chronological record of why the codebase is the way it is -- a sequence of named, purposeful changes each of which makes sense on its own. A poorly maintained commit history is a sequence of "fix", "wip", "more changes", and "final FINAL v2" commits that provide no information about why anything was done.

The standard for commit messages in production engineering is higher than in personal projects. A commit message consists of a subject line of at most 72 characters, an optional blank line, and an optional body that provides context. The subject line uses the imperative mood -- "Add anti-windup to PID controller" rather than "Added anti-windup" or "Adding anti-windup." The body explains why the change was made when that is not obvious from the subject, references relevant issue numbers or ADRs, and notes any non-obvious consequences.

```
Fix signed overflow in velocity saturator

The intermediate computation in compute_velocity() was performed
in int16_t before being multiplied by the PWM scale factor. At
high velocities (>32767 RPM equivalent), this overflowed silently,
producing a large negative duty cycle and driving the motor in
reverse at full power.

Fixed by widening to int32_t before multiplication and narrowing
back to int16_t after clamping to the valid range.

Reproduces in: field report FR-0042 (joint 2 runaway at max speed)
See also: ADR-005 (safe arithmetic patterns)
```

This commit message, read six months from now during a bug investigation, provides everything needed to understand what changed, why, and what real-world problem it addressed.

Tagging releases is equally important. Every version of firmware that is flashed to a robot in the field should correspond to a tagged commit in the repository. The tag name should encode enough information to identify the firmware unambiguously:

```bash
git tag -a v1.4.2 -m "Release 1.4.2 - anti-windup and encoder fault handling"
git push origin v1.4.2
```

When a robot in the field exhibits unexpected behavior, the first question is always "what firmware version is it running?" If the answer maps to a tag, the entire codebase state at that moment is recoverable in seconds. If the answer is "we think it was the build from February," the investigation starts from a much less advantageous position.

### Coding Standards Documents

Every professional robotics team that produces safety-certified software maintains a coding standards document -- a project-specific guide that specifies which rules from MISRA, AUTOSAR, or other guidelines apply, what deviations are pre-approved, what naming conventions are in use, and what the team's specific conventions are for areas not covered by external standards.

This document is not the MISRA handbook. It is a shorter, project-specific document that a new engineer can read in an afternoon and use to answer the question "what does correct code look like on this project?" It serves as the authoritative reference for code review disagreements, for static analysis configuration, and for the deviation register.

A minimal coding standards document covers the following sections. 
- The applicable standard (MISRA C:2012 for C code, AUTOSAR C++14 for C++ code) with a reference to where the full guidelines can be obtained. 
- The C++ standard version -- C++17 in the baseline established in Chapter 5, and which C++20 features are permitted. 
- Pre-approved deviations that apply across the entire codebase, such as the HAL pointer cast deviations from Chapter 6.
- Naming conventions for files, classes, functions, variables, and constants. The required Doxygen documentation level for public interfaces.
- The testing requirements: minimum coverage thresholds, required test types for safety-classified components. 
- The code review requirements: minimum reviewer count, independence requirements for ASIL-classified changes.

The document itself should be version-controlled in the repository alongside the code it governs. Changes to the document go through the same review process as code changes. A team that can change its coding standards without review has coding standards in name only.

### The Complete Workflow

Pulling together every chapter of this module, the complete professional development workflow for a robotics software change looks like this.

1. A developer working on a new feature or a bug fix creates a branch from `develop`. They write the code following the MISRA and AUTOSAR guidelines from Chapter 2, with the defensive patterns from Chapter 4, using the appropriate modern C++ features from Chapter 5, following the structural conventions from Chapter 7, with Doxygen documentation from Chapter 8. If the change involves a significant design decision, they write or update an ADR.

2. Before pushing, they run the static analysis tools from Chapter 6 locally: `cppcheck` for MISRA compliance, `clang-tidy` for modern C++ practices, and address all findings that are not pre-approved deviations. They run the host-machine unit tests from Chapter 9 and verify that coverage has not decreased. For changes to real-time code paths, they run the timing measurements on the STM32F4 target.

3. They push the branch and open a pull request with a description that answers what, why, and how. The CI pipeline from Chapter 6 runs automatically: build with warnings as errors, static analysis, unit tests, coverage check, and documentation generation. If any stage fails, the pull request cannot be merged until it is fixed.

4. Reviewers apply the checklist from this chapter. They verify correctness, contract compliance, error handling, resource management, coding standard compliance, test coverage, documentation, and timing impact. They leave comments as questions or suggestions rather than mandates -- the author knows the context better than the reviewer, and the review is a conversation, not an audit. The author addresses comments, pushes updates, and the CI pipeline reruns. When the reviewers are satisfied and the pipeline is green, the change is merged to `develop`.

5. Periodically -- typically aligned with a release cycle -- the team reviews the technical debt register and allocates capacity to address the highest-priority items. Release candidates are tagged, flashed to hardware for system-level validation, and the tag is promoted to a release tag when validation passes. The release tag, the deviation register, the ADR log, and the generated Doxygen documentation together constitute the engineering artifact that would be submitted to a safety auditor if certification were required.

### Closing Thoughts

The progression through this module has been deliberate. It began with the question of why production code looks different from amateur code, and the answer turned out to be multi-layered. Production code looks different because it is written with an explicit contract for every function, because it treats the full C++ language as too large and too dangerous to use unconstrained, because it is verified by tools that operate on the code's structure rather than just its runtime behavior, because it is organized to be testable independently of the hardware it runs on, because its design decisions are recorded so they can be understood and challenged by future engineers, and because it is developed through a process that transmits knowledge and catches mistakes before they reach deployed systems.

None of the individual techniques in this module are beyond the reach of an undergraduate student who has a working understanding of C++. The mental model they require is not advanced. What they require is discipline -- the habit of applying them consistently rather than only when it seems particularly important. A precondition check that is written sometimes is a precondition check that is missing exactly when it would have caught the bug.

The STM32F4 Discovery board on your desk is a capable piece of hardware. It runs at 168 MHz, has a hardware floating point unit, and is used in real industrial and robotics applications. The gap between the code you write for it today and the code that ships in a production robotics system is not a gap in processor capability or language features. It is a gap in the discipline with which those features are applied. This module has tried to make that discipline concrete, traceable to reasons, and immediately applicable.

The field rewards engineers who write code that other engineers can trust.
