Most embedded software engineers remember this moment clearly. It is the moment something failed in the field, without any obvious cause. A motor may have stalled under load, or it may be a sensor fusion pipeline that produced plausible but wrong estimates after several hours of operation. What a about navigation system that behaves perfectly in the lab but catastrophically fails in a warehouse with slightly different lighting?

There's usually a common defence that a new engineer often jumps to: The code worked. It passed every test the development team wrote. It ran correctly hundreds of times before it didn't.  

It failed, because production software operates under a different contract than software written to pass a demo.

This chapter is about that contract. 

### What Amateur Code Optimizes For

When you are learning to program, the question you are constantly answering is: _does it work?_ You write a function, you test it with a few inputs, it produces the right output, and you move on. This is the correct way to learn. It builds intuition quickly and keeps the feedback loop tight.

But this mindset carries a hidden assumption -- that the environment your code runs in is essentially the same as the environment you tested it in. Same inputs, same timing, same hardware, same temperature, same load on the system. In a classroom or a personal project, that assumption is usually fine.

In a production robotics system, that assumption is almost never true.

Keep in mind that production software runs on hardware that:
- Heats up over a shift.
- Receives EMI corrupted sensor data.
- Runs for months without a restart, accumulating state in ways no unit test ever exercises.

Amateur code asks: _does it work?_ Production code asks: _can I prove it works, and will it keep working?_

There's an important shift in how we think about systems here. We're going from demonstrating correctness to systematically guaranteeing it, and that's what this entire module is about.

### A Motivating Failure

Consider a simplified motor controller, the kind you might write early in a robotics project. Its job is to read a target velocity, compute a PWM duty cycle, and write that value to a hardware register.

```c
#include <stdint.h>

#define PWM_MAX 1000

int16_t compute_duty_cycle(int16_t target_rpm, int16_t max_rpm) {
    int16_t duty = (target_rpm / max_rpm) * PWM_MAX;
    return duty;
}

void set_motor_pwm(int16_t duty_cycle) {
    // Write to hardware register
    PWM_REGISTER = duty_cycle;
}
```

This code looks reasonable. The logic is simple, the variable names are clear, and if you call `compute_duty_cycle(500, 1000)` you get back 500, which is half of `PWM_MAX`. It works in your test.

Now consider what happens when `target_rpm` is 499 and `max_rpm` is 1000. Integer division in C truncates toward zero, so `499 / 1000` evaluates to 0, not 0.499. The duty cycle is zero. The motor does not move. No error is reported. The system continues running, cheerfully sending a zero PWM signal and you wonder why the robot isn't going anywhere.

Now consider what happens when a bug elsewhere in the system passes `max_rpm` as zero. Division by zero on an embedded processor does not throw a polite exception. On ARM Cortex-M processors (the family that includes the STM32F4 on my desk) integer division by zero triggers a HardFault exception. If your fault handler does nothing useful, the system either hangs or resets. In a moving robot, either outcome can be dangerous.

And now consider a subtler problem. Suppose `target_rpm` is 32767 -- the maximum value of a signed 16-bit integer -- and `max_rpm` is 1000. The intermediate expression `(target_rpm / max_rpm) * PWM_MAX` is `32 * 1000`, which is 32000. That fits in an `int16_t`, so no overflow. But if the computation were written slightly differently, or if a colleague changes `PWM_MAX` to 2000 to support a higher-resolution timer, that multiplication overflows, producing a large negative duty cycle that drives the motor in the wrong direction at full power.

None of these are exotic edge cases. They are the kinds of bugs that exist in codebases today, in systems that are currently deployed.

### The Cost of Getting It Wrong

In consumer software, a bug typically means a crash report and a patch update. In robotics, the costs are different in kind, not just degree.

A mobile robot navigating a factory floor that receives a wrong velocity command can collide with a person or piece of equipment. A robotic arm with a corrupted joint angle target can exceed its mechanical limits and destroy itself; or worse, injure someone nearby. A surgical robotics system with a timing bug in its control loop can behave erratically during a procedure.

Even in scenarios where physical safety is not directly at stake, the cost model is severe. Field debugging of embedded systems is expensive and slow. Reproducing intermittent failures on hardware that is deployed in a remote location is sometimes impossible. A firmware update cycle for a fleet of industrial robots can take weeks to validate and deploy safely. The bug you introduced in an afternoon can cost months of engineering time and significant money to diagnose and fix.

This is not meant to frighten you away from robotics software. It is meant to explain why the industry developed the tools, standards, and processes covered in this module. Every rule in MISRA, every discipline in defensive programming, every stage in a static analysis pipeline -- each one exists because someone, somewhere, paid the cost of not having it.

### Reliability Is a Design Property

One of the most important shifts in thinking that separates junior from senior embedded engineers is understanding that reliability is not something you add to code after it is written. It is a property that has to be designed in from the beginning.

This is analogous to mechanical engineering. You cannot build a bridge out of the wrong material and then make it structurally sound by painting it carefully. The material choice, the joint design, the load calculations -- these determine whether the structure is reliable. The finishing touches matter, but they cannot compensate for poor foundations.

In software, the equivalent of choosing the right material is choosing the right language subset. C and C++ are extraordinarily powerful languages, but they expose enormous surface area for undefined behavior, implementation-defined behavior, and unintentional complexity. The full language is too large and too dangerous for safety-critical work. This is why the industry converged on standardized subsets:  MISRA C, MISRA C++, and AUTOSAR C++14 -- that constrain the language to a region where behavior is predictable and analyzable.

- The equivalent of load calculations is **static analysis** -- automated tools that examine your code before it ever runs and identify classes of defects that are invisible to compilers and invisible to testing.

- The equivalent of proper joint design is defensive programming -- a set of coding patterns that make your code resilient to the unexpected, that fail loudly and informatively rather than silently and dangerously.

We will cover all of these in depth across the chapters that follow. But before we do, it is worth naming the verification mindset explicitly, because it underlies all of them.

### The Verification Mindset

The verification mindset asks a specific question about every piece of code: _under what conditions can I guarantee this behaves correctly?_

Not "I tested it and it worked." Not "it seems fine." But: what are the precise preconditions that must hold for this function to produce a correct result? 
What are the postconditions -- the guarantees it makes when it returns? What happens if a precondition is violated?

This is the intellectual framework behind formal methods, design-by-contract, and the safety standards you will encounter in this module. You do not need to write formal proofs for every function you ship. But you do need to think about code the way someone who writes formal proofs does -- with explicit assumptions, explicit guarantees, and explicit handling of the cases where those assumptions break down.

Here is what that looks like applied to the motor controller example from earlier:

```c
#include <stdint.h>
#include <assert.h>

#define PWM_MAX 1000

/**
 * @brief Compute PWM duty cycle from target and maximum RPM.
 *
 * @pre max_rpm > 0
 * @pre target_rpm >= 0 && target_rpm <= max_rpm
 * @post return value is in range [0, PWM_MAX]
 */
int16_t compute_duty_cycle(int16_t target_rpm, int16_t max_rpm) {
    assert(max_rpm > 0);
    assert(target_rpm >= 0 && target_rpm <= max_rpm);

    /* Use 32-bit intermediate to avoid overflow */
    int32_t duty = ((int32_t)target_rpm * PWM_MAX) / max_rpm;

    assert(duty >= 0 && duty <= PWM_MAX);
    return (int16_t)duty;
}
```

This version is not dramatically longer. But notice what changed. 
- The integer truncation bug is fixed by reordering the operations -- multiply before dividing. 
- The overflow risk is eliminated by performing the intermediate computation in a wider type and making the cast explicit. 
- The division-by-zero risk is caught at the point of entry by an explicit precondition check. And the behavior of the function is now _documented_  -- a future maintainer reading this code knows immediately what inputs are valid and what the function guarantees about its output.

Assertions are not the only tool for enforcing preconditions, and they have important limitations in production embedded systems -- we will discuss when to use them and when to replace them with something more robust in Chapter 4. But the thinking they represent is exactly right: _state your assumptions explicitly, and check them._
