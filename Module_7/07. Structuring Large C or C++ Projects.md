There is a particular kind of confusion that sets in when you first open a professional robotics codebase. The file count is in the hundreds. The directory structure has layers you did not expect. There are abstraction boundaries that seem unnecessary until the moment you understand why they exist. Code that looks like it does the same thing is written differently in different parts of the project, and that difference turns out to be intentional.

This chapter explains the organizational principles behind that structure. Understanding them not only helps you navigate unfamiliar codebases -- it gives you the vocabulary and the reasoning to make good structural decisions when you are the one writing the code from scratch.

### The Core Problem: Managing Complexity at Scale

A single-file embedded program is easy to understand in its entirety. Every variable, every function, every dependency is visible at once. As a project grows, this totality of visibility becomes impossible to maintain. A robotics system with sensor drivers, a communication stack, a motion planner, a control loop, a state machine, and a configuration system cannot fit in a single developer's working memory simultaneously.

The response to this problem is decomposition -- breaking the system into pieces that can be understood independently. But decomposition alone is not enough. The way the pieces connect to each other matters as much as the pieces themselves. A system decomposed into fifty modules that all depend on each other in arbitrary ways is not simpler than a monolith. It is a monolith with extra steps.

The discipline of managing these connections -- deciding what each module knows about, what it is allowed to depend on, and what it exposes to the rest of the system -- is called physical design. The term comes from John Lakos's foundational work _Large-Scale C++ Software Design_, published in 1996 and still relevant today. The core insight is that the physical structure of a codebase -- how files are organized, what includes what, which libraries depend on which — determines the long-term maintainability of the system as much as the quality of the code within those files.

There's even commercial platforms that help you create dependency graphs for your C++ project. Check out CppDepend: https://www.cppdepend.com/documentation/cppdepend-dependency-graph

### Components and Dependency Direction

The fundamental unit of physical design is the component. In a C/C++ codebase, a component is a header file and its corresponding implementation file, treated as a pair. The header declares the public interface -- the types, functions, and constants that other components are allowed to use. The implementation file defines the behavior behind that interface. Nothing outside the component should depend on anything inside the implementation file.

The rule that makes components composable is acyclic dependency direction. If component A depends on component B -- meaning A's header includes B's header -- then B must not depend on A, directly or indirectly. Dependency cycles make it impossible to build, test, or understand components in isolation. They mean that to understand A you must understand B, and to understand B you must understand A, and you can never get a foothold.

In practice, dependency direction should flow in one direction through the codebase: from high-level application code down toward low-level hardware drivers, never the other way around. A motion planner may depend on a kinematics library. A kinematics library may depend on a linear algebra library. A linear algebra library should not depend on a motion planner.

This principle has a direct consequence for how you manage `#include` directives. Every `#include` in a header file is a dependency declaration -- it says that anything using your component must also have access to everything you included. Header files should include only what they genuinely need for their public interface. Everything else belongs in the implementation file. This discipline, sometimes called include-what-you-use, keeps the dependency graph as sparse as possible.

The `Include What You Use` tool ([https://include-what-you-use.org](https://include-what-you-use.org)) automates this analysis and is worth integrating alongside clang-tidy in a CI pipeline for larger projects.

### A Practical Directory Structure for Robotics Projects

Abstract principles are easier to grasp with a concrete example. A typical mid-sized robotics project, -- say, a six-axis robotic arm controller -- might be organized like this:

```
robot_arm/
├── CMakeLists.txt
├── .clang-tidy
├── misra_suppressions.txt
│
├── hal/                        # Hardware Abstraction Layer
│   ├── include/
│   │   └── hal/
│   │       ├── uart.hpp
│   │       ├── gpio.hpp
│   │       ├── encoder.hpp
│   │       └── pwm.hpp
│   ├── src/
│   │   ├── stm32f4/            # STM32-specific implementations
│   │   │   ├── uart_stm32.cpp
│   │   │   ├── gpio_stm32.cpp
│   │   │   └── pwm_stm32.cpp
│   │   └── sim/                # Simulation implementations
│   │       ├── uart_sim.cpp
│   │       └── gpio_sim.cpp
│   └── CMakeLists.txt
│
├── drivers/                    # Device drivers built on HAL
│   ├── include/
│   │   └── drivers/
│   │       ├── encoder_driver.hpp
│   │       └── motor_driver.hpp
│   ├── src/
│   │   ├── encoder_driver.cpp
│   │   └── motor_driver.cpp
│   └── CMakeLists.txt
│
├── control/                    # Control algorithms
│   ├── include/
│   │   └── control/
│   │       ├── pid_controller.hpp
│   │       └── trajectory.hpp
│   ├── src/
│   │   ├── pid_controller.cpp
│   │   └── trajectory.cpp
│   └── CMakeLists.txt
│
├── kinematics/                 # Robot kinematics
│   ├── include/
│   │   └── kinematics/
│   │       └── forward_kinematics.hpp
│   ├── src/
│   │   └── forward_kinematics.cpp
│   └── CMakeLists.txt
│
├── app/                        # Application layer
│   ├── main.cpp
│   └── CMakeLists.txt
│
└── tests/                      # Unit and integration tests
    ├── test_pid_controller.cpp
    ├── test_kinematics.cpp
    └── CMakeLists.txt
```

Several things about this structure are worth noting explicitly. Each module has its own `include/` subdirectory, and headers are nested under a directory named after the module --`hal/uart.hpp` rather than just `uart.hpp`. This means includes in source files look like `#include "hal/uart.hpp"` rather than `#include "uart.hpp"`, which makes the dependency explicit at the point of use and prevents name collisions between headers from different modules.

The `hal/` directory is split between interface and implementation, with platform-specific implementations in named subdirectories. This is the Hardware Abstraction Layer pattern, which deserves its own detailed treatment.

The dependency direction flows downward: `app` depends on `control` and `kinematics`, `control` depends on `drivers`, `drivers` depend on `hal`. Nothing in `hal` depends on `control` or `app`. The dependency graph is a directed acyclic graph, and each layer can be built, understood, and tested without knowledge of the layers above it.

### The Hardware Abstraction Layer

The Hardware Abstraction Layer, universally abbreviated as HAL, is the boundary between code that knows about specific hardware and code that does not. Everything below the HAL is platform-specific. Everything above it is portable.

The motivation for this separation is practical on multiple fronts. Code above the HAL can be compiled and tested on a development machine without requiring the target hardware to be present. When the project moves to a different microcontroller -- which happens more often than anyone plans for at the start of a project -- only the HAL implementation needs to change. Simulation environments can provide a HAL implementation that models hardware behavior without needing physical hardware at all.

In C++, the HAL is typically expressed as a set of abstract base classes or interfaces. A concrete implementation for each supported platform inherits from the interface and provides the platform-specific behavior. The rest of the codebase depends only on the interface, never on the concrete implementation.

Here is a minimal UART interface for the robotic arm controller:

```cpp
// hal/include/hal/uart.hpp
#pragma once
#include <cstdint>
#include <cstddef>

namespace hal {

class IUart {
public:
    virtual ~IUart() = default;

    virtual bool transmit(const uint8_t* data,
                          size_t         length,
                          uint32_t       timeout_ms) = 0;

    virtual bool receive(uint8_t*  buffer,
                         size_t    length,
                         uint32_t  timeout_ms) = 0;

    virtual bool is_ready() const = 0;
};

}  // namespace hal
```

The STM32-specific implementation provides the concrete behavior:

```cpp
// hal/src/stm32f4/uart_stm32.cpp
#include "hal/uart.hpp"
#include "usart.h"   // STM32 HAL header

namespace hal {

class UartStm32 : public IUart {
public:
    explicit UartStm32(UART_HandleTypeDef* handle)
        : handle_(handle) {}

    bool transmit(const uint8_t* data,
                  size_t         length,
                  uint32_t       timeout_ms) override {
        return HAL_UART_Transmit(handle_,
                                 const_cast<uint8_t*>(data),
                                 static_cast<uint16_t>(length),
                                 timeout_ms) == HAL_OK;
    }

    bool receive(uint8_t*  buffer,
                 size_t    length,
                 uint32_t  timeout_ms) override {
        return HAL_UART_Receive(handle_,
                                buffer,
                                static_cast<uint16_t>(length),
                                timeout_ms) == HAL_OK;
    }

    bool is_ready() const override {
        return handle_->gState == HAL_UART_STATE_READY;
    }

private:
    UART_HandleTypeDef* handle_;
};

}  // namespace hal
```

And the simulation implementation provides behavior suitable for unit testing on a development machine:

```cpp
// hal/src/sim/uart_sim.cpp
#include "hal/uart.hpp"
#include <vector>
#include <cstring>

namespace hal {

class UartSim : public IUart {
public:
    bool transmit(const uint8_t* data,
                  size_t         length,
                  uint32_t       /*timeout_ms*/) override {
        tx_buffer_.insert(tx_buffer_.end(), data, data + length);
        return true;
    }

    bool receive(uint8_t*  buffer,
                 size_t    length,
                 uint32_t  /*timeout_ms*/) override {
        if (rx_buffer_.size() < length) return false;
        std::memcpy(buffer, rx_buffer_.data(), length);
        rx_buffer_.erase(rx_buffer_.begin(),
                         rx_buffer_.begin() + static_cast<ptrdiff_t>(length));
        return true;
    }

    bool is_ready() const override { return true; }

    /* Test helper: inject data for the next receive call */
    void inject_rx_data(const uint8_t* data, size_t length) {
        rx_buffer_.insert(rx_buffer_.end(), data, data + length);
    }

    const std::vector<uint8_t>& get_tx_buffer() const { return tx_buffer_; }

private:
    std::vector<uint8_t> tx_buffer_;
    std::vector<uint8_t> rx_buffer_;
};

}  // namespace hal
```

Code that uses UART communication takes an `IUart&` reference rather than a concrete type. In production firmware it receives a `UartStm32`. In unit tests it receives a `UartSim`. The code under test is identical in both cases, which means the tests are actually testing the production code path rather than a separate code path written specifically for testing.

### CMake: The Build System in Practice

CMake is the de facto build system for C and C++ projects in the robotics industry. ROS 2 uses it. Most commercial robotics software stacks use it. Understanding CMake well enough to structure a multi-module project and configure it for cross-compilation is a practical necessity.

CMake does not build code directly. It generates build files for another tool -- typically `make` on Linux or Ninja, which is faster and increasingly preferred. You describe your project in `CMakeLists.txt` files, run CMake to generate the build files, and then run the build tool to compile.

A well-structured CMake project has one `CMakeLists.txt` at the root and one in each module directory. The root file sets global properties and adds the module subdirectories. Each module file defines a library target and declares its dependencies.

The root `CMakeLists.txt` for the robotic arm project:

```cmake
cmake_minimum_required(VERSION 3.20)
project(robot_arm VERSION 1.0.0 LANGUAGES CXX C)

# Require C++17 across the entire project
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)   # Disable compiler-specific extensions

# Export compilation database for clang-tidy
set(CMAKE_EXPORT_COMPILE_COMMANDS ON)

# Global warning flags
add_compile_options(
    -Wall
    -Wextra
    -Wpedantic
    -Wshadow
    -Wconversion
    -Wsign-conversion
    -Werror
)

# Add module subdirectories
add_subdirectory(hal)
add_subdirectory(drivers)
add_subdirectory(control)
add_subdirectory(kinematics)
add_subdirectory(app)
add_subdirectory(tests)
```

The `CMAKE_CXX_EXTENSIONS OFF` setting deserves attention. GCC and Clang both support non-standard extensions to C++ that are enabled by default. Disabling them ensures that the code compiles strictly according to the C++17 standard, which is important for portability and for MISRA compliance checking.

The `hal/CMakeLists.txt` shows how a module defines its library and declares what it exposes:

```cmake
# hal/CMakeLists.txt

# Detect build target to select correct implementation
if(CMAKE_CROSSCOMPILING)
    set(HAL_IMPL_DIR src/stm32f4)
else()
    set(HAL_IMPL_DIR src/sim)
endif()

add_library(hal STATIC
    ${HAL_IMPL_DIR}/uart_stm32.cpp
    ${HAL_IMPL_DIR}/gpio_stm32.cpp
    ${HAL_IMPL_DIR}/pwm_stm32.cpp
)

# PUBLIC: consumers of hal also get this include path
target_include_directories(hal
    PUBLIC  include/
    PRIVATE src/
)

# Link against STM32 HAL when cross-compiling
if(CMAKE_CROSSCOMPILING)
    target_link_libraries(hal PUBLIC stm32_hal)
endif()
```

The distinction between `PUBLIC`, `PRIVATE`, and `INTERFACE` in CMake target properties is one of the more important concepts to internalize. A `PUBLIC` include directory is propagated to every target that links against this library -- consumers automatically get the include path without needing to declare it themselves. A `PRIVATE` include directory is used only when compiling this library. `INTERFACE` is used when the library is header-only and does not compile any source files itself.

The `drivers/CMakeLists.txt` shows how a module declares its dependency on `hal`:

```cmake
# drivers/CMakeLists.txt
add_library(drivers STATIC
    src/encoder_driver.cpp
    src/motor_driver.cpp
)

target_include_directories(drivers
    PUBLIC  include/
    PRIVATE src/
)

# Declaring this dependency gives drivers access to hal's PUBLIC include paths
target_link_libraries(drivers PUBLIC hal)
```

With this structure in place, CMake handles the entire dependency graph. Building `app` automatically builds `control`, `drivers`, and `hal` in the correct order, with the correct include paths and link flags for each step.

### Cross-Compilation for ARM Targets

Most robotics projects compile code that runs on a different architecture from the development machine. Writing code on an x86 laptop and deploying it to an ARM Cortex-M4 or an ARM Cortex-A72 is cross-compilation, and managing it correctly is one of the practical challenges of embedded robotics development.

CMake handles cross-compilation through toolchain files -- CMake scripts that specify the compiler, linker, and system libraries for the target architecture. For STM32 targets, the ARM GCC toolchain file looks like this:

```cmake
# toolchains/stm32f4.cmake

set(CMAKE_SYSTEM_NAME Generic)
set(CMAKE_SYSTEM_PROCESSOR arm)

# Specify the cross compiler
set(CMAKE_C_COMPILER   arm-none-eabi-gcc)
set(CMAKE_CXX_COMPILER arm-none-eabi-g++)
set(CMAKE_ASM_COMPILER arm-none-eabi-gcc)
set(CMAKE_OBJCOPY      arm-none-eabi-objcopy)
set(CMAKE_SIZE         arm-none-eabi-size)

# Cortex-M4 with FPU flags
set(CPU_FLAGS
    "-mcpu=cortex-m4 -mthumb -mfpu=fpv4-sp-d16 -mfloat-abi=hard"
)

set(CMAKE_C_FLAGS_INIT   "${CPU_FLAGS}")
set(CMAKE_CXX_FLAGS_INIT "${CPU_FLAGS} -fno-exceptions -fno-rtti")

# Tell CMake not to try to link test executables during configuration
set(CMAKE_TRY_COMPILE_TARGET_TYPE STATIC_LIBRARY)
```

The `-fno-exceptions` and `-fno-rtti` flags are significant. Disabling exceptions removes the stack unwinding machinery from the binary, which reduces code size and eliminates the nondeterministic runtime cost of exception handling -- appropriate for the hard real-time paths discussed in Chapter 5. Disabling RTTI removes the runtime type information tables used by `dynamic_cast` and `typeid`, which also reduces binary size. As noted in Chapter 5, neither of these features should be used in safety-critical real-time code, so disabling them at the compiler level makes that constraint automatic rather than voluntary.

To build for the STM32F4 target:

```bash
cmake -B build/stm32 \
      -DCMAKE_TOOLCHAIN_FILE=toolchains/stm32f4.cmake \
      -DCMAKE_BUILD_TYPE=Release

cmake --build build/stm32
```

To build for the host machine (for running unit tests) omit the toolchain file:

```bash
cmake -B build/host \
      -DCMAKE_BUILD_TYPE=Debug

cmake --build build/host
```

This dual-build capability is one of the most valuable properties of a well-structured robotics project. The same source code, the same CMakeLists, and the same tests run both on your laptop and on the target hardware. The HAL abstraction provides the platform-specific behavior in each case.

### Header Hygiene in Practice

Large projects accumulate header inclusion debt over time. Developers add includes when a new dependency is needed and rarely remove them when the dependency is no longer needed. The result is headers that pull in large portions of the codebase unnecessarily, increasing build times and making dependency relationships misleading.

Three rules applied consistently keep header files clean. First, every header file should include exactly what it needs for its public interface to compile -- no more. If a type is used only as a pointer or reference in a header, a forward declaration is sufficient and a full include is not needed. Second, implementation files should not rely on transitive includes -- if `motor_driver.cpp` needs `<cstdint>`, it should include it directly rather than relying on the fact that some header it includes happens to include it. Third, include guards or `#pragma once` should appear in every header file without exception.

`#pragma once` is not strictly standard C++, but it is supported by every compiler used in robotics development and is simpler and less error-prone than traditional include guards. AUTOSAR Rule A16-2-3 recommends it explicitly.

```cpp
// Correct: #pragma once protects against multiple inclusion
#pragma once

#include <cstdint>
#include "hal/uart.hpp"   // Full include needed: IUart is a base class

// Forward declaration sufficient: MotorConfig is only used as a pointer
class MotorConfig;

namespace drivers {

class MotorDriver {
public:
    explicit MotorDriver(hal::IUart& uart);
    bool configure(const MotorConfig* config);

private:
    hal::IUart& uart_;
};

}  // namespace drivers
```

### Version Control Discipline

A large project without disciplined version control is a large project that cannot be maintained. The structural decisions made in CMakeLists files, the toolchain configurations, the static analysis configurations, the suppression files -- all of these belong in version control alongside the source code, and changes to them should go through the same review process as changes to production code.

Branch strategy in embedded robotics typically follows a pattern where `main` always contains code that has passed all CI checks and can be built and deployed, `develop` is the integration branch where features are combined before being promoted to `main`, and feature branches are short-lived branches where individual changes are developed. Long-lived feature branches accumulate merge conflicts and should be avoided. Commit messages deserve more care than they typically receive. A commit message that says "fix bug" tells a future engineer nothing. A commit message that says "Fix signed overflow in velocity saturator -- intermediate computation was performed in int16_t, causing wrap-around at high velocities. Widened to int32_t before multiplication." tells them exactly what changed, why, and what the consequence of the old behavior was. Six months later, when a similar issue appears in a different module, that commit message is the fastest path to understanding what the correct fix looks like.
