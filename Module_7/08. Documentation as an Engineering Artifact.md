Documentation is the part of software development that almost everyone agrees is important and almost everyone underinvests in. The reasons are understandable. Writing code produces immediate, visible results. Writing documentation produces something whose value is diffuse and deferred -- it pays off when a new team member joins, when a bug needs to be traced six months after the fact, when a safety auditor asks for evidence that a design decision was deliberate. These payoffs are real, but they are invisible at the moment the documentation is written.

In production robotics software, this calculus changes. Documentation is not a courtesy extended to future maintainers. It is part of the engineering artifact. A function whose preconditions are not documented has an incomplete specification. An architectural decision that is not recorded is a decision that will be relitigated every time a new engineer encounters the code and wonders why it was done that way. In safety-certified software, documentation is often a contractual requirement -- the evidence package submitted to a certification body includes design documents, interface specifications, and records of design decisions alongside the source code itself.

This chapter covers three layers of documentation that together constitute a complete approach: API documentation with Doxygen, architectural documentation through structured prose and Architecture Decision Records, and the discipline of writing comments that add genuine value rather than restating what the code already says.

### What Documentation Is Actually For

Before getting into tools and formats, it is worth being precise about what good documentation accomplishes, because the answer shapes every decision about what to write.

API documentation tells a caller what a function does, what inputs it accepts, what it guarantees about its outputs, and what can go wrong. Its audience is the developer writing code that uses your component. It should make the header file self-sufficient -- a caller should be able to use your component correctly without reading the implementation.

Implementation comments explain why the code does what it does. Their audience is the developer who will modify the code in the future, which is often you, six months from now, having forgotten everything about the context in which you wrote it. The question they answer is not "what does this code do" (the code answers that) but "why does it do it this way, and what alternatives were considered and rejected." Architectural documentation explains how the system is organized, how the major components relate to each other, and why the structure is the way it is. Its audience is anyone who needs to understand the system at a level above individual functions -- a new team member, a technical lead reviewing the design, a safety auditor verifying that the system meets its requirements.

Each layer has a different tool and a different style. Mixing them -- writing architectural rationale in Doxygen comments, or writing API documentation in a separate prose document, produces something that is harder to find and harder to maintain than keeping each layer in its proper place.

### Doxygen: API Documentation That Lives in the Code

Doxygen recognizes comments that begin with `/**` or `///`. Within those comments, special commands prefixed with `@` or `\` annotate specific aspects of the documented entity. The commands used most frequently in robotics code are:

- `@brief` provides a single-sentence summary of what a function or class does. It appears in overview tables and in tooltips in IDE integrations. It should be concise enough to read in one line.

- `@param` documents a function parameter. It takes the parameter name and a description. For parameters with constraints -- a value that must be positive, a pointer that must not be null -- those constraints belong here, not in a separate note.

- `@return` documents what the function returns, including the meaning of special return values like error codes.

- `@pre` and `@post` document preconditions and postconditions explicitly, connecting the Doxygen comment to the design-by-contract approach from Chapter 4.

- `@note` adds supplementary information that does not fit neatly into the other categories — performance characteristics, thread safety, hardware dependencies.

- `@see` cross-references related functions or types.

Here is a well-documented version of the PID controller from Chapter 4, showing how these commands work together to produce complete interface documentation:

```cpp
// control/include/control/pid_controller.hpp
#pragma once

#include <cstdint>

namespace control {

/**
 * @brief Discrete-time PID controller with output saturation.
 *
 * Implements a standard proportional-integral-derivative controller
 * using the backward Euler method for integration and first-order
 * finite differences for the derivative term. Output is clamped to
 * the range [-output_limit, output_limit] to prevent actuator
 * saturation from propagating into the integrator.
 *
 * @note This class is not thread-safe. If the controller is shared
 *       between tasks, the caller is responsible for synchronization.
 *
 * @note Integral windup is not prevented beyond output clamping.
 *       For applications with prolonged saturation, consider adding
 *       anti-windup logic via reset_integrator().
 */
class PidController {
public:
    /**
     * @brief Construct a PID controller with given gains and output limit.
     *
     * @param kp            Proportional gain. Must be >= 0.
     * @param ki            Integral gain. Must be >= 0.
     * @param kd            Derivative gain. Must be >= 0.
     * @param output_limit  Maximum absolute output value. Must be > 0.
     *
     * @pre output_limit > 0.0f
     */
    PidController(float kp,
                  float ki,
                  float kd,
                  float output_limit);

    /**
     * @brief Compute the next control output.
     *
     * Updates the controller state and returns the control signal
     * for the current timestep. The output is clamped to
     * [-output_limit, output_limit] before returning.
     *
     * @param setpoint     Desired value.
     * @param measurement  Current measured value.
     * @param dt_s         Elapsed time since last call, in seconds.
     *                     Must be > 0.
     *
     * @return Control output in the range [-output_limit, output_limit].
     *
     * @pre dt_s > 0.0f
     * @post |return value| <= output_limit
     *
     * @see reset_integrator()
     */
    [[nodiscard]] float update(float setpoint,
                               float measurement,
                               float dt_s);

    /**
     * @brief Reset the integrator and derivative state to zero.
     *
     * Should be called when the controller is re-enabled after a
     * period of inactivity to prevent integrator windup from a
     * previous run from causing a large initial output spike.
     */
    void reset_integrator();

    /**
     * @brief Return the current value of the integral accumulator.
     *
     * Intended for diagnostics and logging. The returned value is
     * in the same units as (error * time).
     *
     * @return Current integral accumulator value.
     */
    [[nodiscard]] float get_integral() const;

private:
    float kp_;
    float ki_;
    float kd_;
    float output_limit_;
    float integral_;
    float prev_error_;
};

}  // namespace control
```

Reading this header, a caller knows exactly what the class does, what constraints the constructor parameters must satisfy, what `update()` guarantees about its return value, and what to do when resuming control after a pause. They do not need to read the implementation. That is the standard a production API header should meet.

#### Documenting Error-Returning Functions

Functions that return status codes or `std::optional` values require particular care in their documentation, because the caller's behavior should differ based on the return value and that difference needs to be explicit.

```cpp
/**
 * @brief Read the current encoder position for a joint.
 *
 * Queries the hardware encoder register for the specified joint
 * and returns the raw tick count. Returns an absent value if the
 * joint ID is out of range or if the encoder reports a fault
 * condition.
 *
 * @param joint_id  Zero-based joint index. Valid range: [0, NUM_JOINTS).
 *
 * @return Encoder tick count if successful, std::nullopt if:
 *         - joint_id >= NUM_JOINTS
 *         - Encoder fault bit is set in the status register
 *
 * @note The tick count wraps at INT32_MAX and INT32_MIN. Callers
 *       performing velocity estimation must handle this wrap-around.
 *
 * @see EncoderDriver::clear_fault()
 */
[[nodiscard]] std::optional<int32_t> read_position(uint8_t joint_id) const;
```

The `@return` description lists the specific conditions under which `std::nullopt` is returned. This matters because different failure conditions may warrant different responses in the caller -- a joint ID error indicates a programming mistake, while an encoder fault indicates a hardware condition that needs to be handled at runtime.

#### Configuring Doxygen

A `Doxyfile` at the project root configures Doxygen's behavior. The most important settings for a robotics project are:

```
# Doxyfile (selected settings)

PROJECT_NAME           = "Robot Arm Controller"
PROJECT_VERSION        = "1.0.0"
OUTPUT_DIRECTORY       = docs/

# Parse all headers in the include directories
INPUT                  = hal/include \
                         drivers/include \
                         control/include \
                         kinematics/include

RECURSIVE              = YES
FILE_PATTERNS          = *.hpp *.h

# Extract documentation even for undocumented entities
# (useful for seeing what still needs documenting)
EXTRACT_ALL            = NO
EXTRACT_PRIVATE        = NO
WARN_IF_UNDOCUMENTED   = YES

# Generate call graphs if Graphviz is installed
HAVE_DOT               = YES
CALL_GRAPH             = YES
CALLER_GRAPH           = YES

# HTML output
GENERATE_HTML          = YES
GENERATE_LATEX         = NO

# Treat @pre and @post as named sections
ALIASES               += "pre=@par Preconditions:"
ALIASES               += "post=@par Postconditions:"
```

The `WARN_IF_UNDOCUMENTED = YES` setting is particularly valuable in a CI context. Running Doxygen with this setting enabled and treating warnings as errors means that an undocumented public function fails the CI build, creating the same enforcement mechanism for documentation completeness that `-Werror` creates for compiler warnings.

Adding a Doxygen step to the CI pipeline from Chapter 6:

```yaml
- name: Generate and verify documentation
  run: |
    doxygen Doxyfile 2>&1 | tee doxygen_warnings.txt
    if [ -s doxygen_warnings.txt ]; then
      echo "Doxygen warnings found:"
      cat doxygen_warnings.txt
      exit 1
    fi
```

### Writing Comments That Add Value

Doxygen handles API documentation. Implementation comments ( the ones inside function bodies) serve a different purpose and follow different rules.

The most common failure mode for implementation comments is restating what the code already says. These comments add noise without adding information and make the code harder to read rather than easier:

```cpp
// Bad: comment restates the code
int32_t velocity = 0;   // Set velocity to zero
encoder_.reset();       // Reset the encoder
state_ = State::IDLE;   // Set state to IDLE
```

A reader who can read C++ does not need these comments. They slow down reading and create maintenance burden -- every time the code changes, the comment must change too, and when it does not, the comment becomes actively misleading.

The comments worth writing are the ones that answer a question the code cannot answer by itself. Why was this particular algorithm chosen over the obvious alternative? What hardware behavior made this workaround necessary? What does this magic constant mean and where did it come from?

```cpp
// Good: explains why, not what
// Velocity is zeroed before encoder reset because the encoder
// hardware latches the last velocity sample. If we reset the
// encoder while a non-zero velocity is latched, the first
// velocity reading after reset will be wrong by that amount.
// See STM32F4 encoder timer errata ES0206, section 2.3.
velocity_ = 0;
encoder_.reset();
state_ = State::IDLE;
```

This comment gives a future maintainer everything they need to understand the ordering constraint and verify that it still applies if the code is changed. It references the source of the knowledge (a specific errata document) so that the constraint can be traced back to authoritative documentation rather than treated as an unexplained ritual.

Numerical constants in control and robotics code almost always need comments explaining their origin:

```cpp
// Conversion factor: encoder is 4096 ticks/revolution (quadrature,
// 1024 CPR sensor), gear ratio is 50:1, output in radians.
// Factor = (2 * pi) / (4096 * 50) = 3.0680e-5 rad/tick
constexpr float TICKS_TO_RAD = 3.0680e-5f;
```

Every non-trivial codebase contains decisions that were made for good reasons that are not visible in the code itself. Why does the HAL use virtual dispatch rather than a template-based policy design? Why is the control loop frequency 1 kHz rather than 500 Hz or 2 kHz? Why does the project use fixed-capacity containers from ETL rather than a custom implementation?

These decisions are made once, usually after careful consideration of alternatives, and then become invisible. A new engineer joining the team encounters the code, sees the decision, and either trusts that it was made for good reasons or spends time relitigating it -- implementing a prototype of the alternative, discovering why it does not work, and arriving back at the same conclusion the original authors reached. This is expensive.

Architecture Decision Records, commonly called ADRs, are a lightweight format for recording these decisions in a way that travels with the codebase. Each ADR is a short document, typically a single markdown file, that records one decision: the context that made the decision necessary, the options that were considered, the decision that was made, and the consequences -- both the benefits and the tradeoffs accepted.

ADRs are stored in a `docs/decisions/` directory in the repository, numbered sequentially. They are never modified after they are accepted -- if a decision is revisited and changed, a new ADR is written that supersedes the old one. The old ADR remains in the repository as a historical record.

A concrete ADR for the HAL design decision from Chapter 7:

```markdown
# ADR-001: Hardware Abstraction Layer Using Virtual Dispatch

## Status
Accepted — 2024-01-15

## Context
The robot arm controller must run on two targets: the STM32F407
production hardware and a simulation environment on x86 development
machines. Unit tests for the driver and control layers must run
on development machines without requiring physical hardware.

We need a mechanism to swap hardware-specific implementations
without modifying the code that depends on them.

## Options Considered

### Option 1: Virtual dispatch (abstract base class)
Define an interface as an abstract base class. Concrete
implementations inherit from it. Callers hold a reference
to the interface.

Pros: Standard C++ idiom, well-understood, works cleanly
with dependency injection, compatible with most static
analysis tools.

Cons: Virtual function call overhead (~1-5ns per call on
Cortex-M4). Cannot be inlined.

### Option 2: Template-based policy design
Define implementations as template parameters. The compiler
instantiates separate code for each target.

Pros: Zero runtime overhead, fully inlineable.

Cons: Requires template parameters to propagate through the
entire call stack, significantly increasing compilation
complexity and reducing readability. Makes dependency
injection in tests more complex.

### Option 3: Compile-time #ifdef selection
Use preprocessor conditionals to select implementations.

Pros: Zero overhead.

Cons: Cannot swap implementations at runtime (prevents
hardware-in-the-loop simulation). Violates AUTOSAR Rule
A16-0-1 (no business logic in preprocessor). Poor
testability.

## Decision
Option 1: Virtual dispatch.

The virtual call overhead is acceptable. HAL functions are
called at the driver layer, not in the inner control loop.
The highest-frequency HAL calls are encoder reads at 1 kHz,
where a 5ns overhead is negligible relative to the 1ms
period.

The testability and readability benefits of virtual dispatch
outweigh the performance cost at this layer. Performance-
critical code in the control layer does not call HAL
functions directly.

## Consequences
- HAL interface headers must be available on all build targets.
- Concrete implementations are selected at link time via CMake
  target configuration.
- The control layer remains free of HAL dependencies and can
  be tested without hardware.
- If profiling reveals that virtual dispatch overhead is
  significant (unlikely), this decision can be revisited
  with ADR-002.
```

### Module-Level Documentation

Between individual function documentation and architectural ADRs, there is a middle layer that is often missing: module-level documentation that explains the purpose, responsibilities, and boundaries of each component.

A module-level document lives in a `README.md` at the root of each module directory. It is not a comprehensive reference -- Doxygen handles that. It is a narrative introduction that gives a new reader enough context to understand what the module is for and how to use it before diving into the headers.

A good module README for the control layer might look like this:

```markdown
# control

Discrete-time control algorithms for the robot arm joints.

## Overview

This module implements the PID controllers and trajectory
interpolation used in the joint-level control loop. It operates
at 1 kHz and is designed to run in the real-time task context.
No dynamic memory allocation occurs within this module after
initialization.

## Dependencies

- `kinematics`: Forward kinematics used to convert joint-space
  trajectories to Cartesian space for logging.
- No HAL dependencies: all hardware interaction is done through
  the `drivers` layer.

## Usage

Construct a PidController with the desired gains and output limit.
Call update() at each control loop tick with the current setpoint
and measurement. Call reset_integrator() whenever the controller
is re-enabled after an inactive period.

See docs/decisions/ADR-003.md for the choice of backward Euler
integration over trapezoidal integration.

## Thread Safety

PidController is not thread-safe. In the multi-task configuration,
each joint has its own controller instance accessed only from its
own control task. No synchronization is required.

## Real-Time Constraints

All functions in this module complete in bounded time. The worst-
case execution time of PidController::update() has been measured
at 4.2 microseconds on the STM32F407 at 168 MHz with the FPU
enabled. This is within the 50 microsecond budget allocated to
the control computation in the 1 kHz loop.
```

The worst-case execution time entry is worth highlighting. In a safety-certified project, timing measurements like this belong in the documentation and should be updated whenever the implementation changes. A future engineer who optimizes the PID implementation needs to know that the 4.2 microsecond figure exists, where it was measured, and that they need to re-measure after their change.

### Documentation in the STM32CubeIDE Workflow

STM32CubeIDE does not have built-in Doxygen support in the way that Qt Creator or CLion do, but the integration is straightforward to set up manually.

Install Doxygen from [https://www.doxygen.nl/download.html](https://www.doxygen.nl/download.html) and, optionally, Graphviz from [https://graphviz.org/download](https://graphviz.org/download) for call graph generation. Place the `Doxyfile` at the project root.

To add a Doxygen build step in CubeIDE, right-click the project and select Properties. Navigate to C/C++ Build → Build Steps. In the Post-build steps field, add:

```
doxygen ${ProjDirPath}/Doxyfile
```

This runs Doxygen automatically after each successful build, keeping the generated documentation in sync with the source. The output appears in the `docs/` directory, and opening `docs/html/index.html` in a browser gives the full API reference.

For the fault diagnostics project from Chapter 4, adding Doxygen comments to `fault_handler.c` and the HAL interface demonstrates the immediate payoff. The `HardFault_Handler` function and the fault status register definitions become self-documenting, with cross-references between the handler and the register descriptions making the diagnostic output easy to interpret:

```c
/**
 * @brief HardFault exception handler with diagnostic register dump.
 *
 * Reads the Cortex-M4 fault status registers and transmits a
 * human-readable diagnostic message over USART2 before halting.
 * Intended for development and debugging. In production firmware,
 * replace the infinite loop with a safe-state entry and watchdog
 * reset.
 *
 * @note This handler must replace the default HardFault_Handler
 *       in stm32f4xx_it.c. See project setup instructions in
 *       Chapter 4 of the embedded systems course notes.
 *
 * @see CFSR register description: ARM DDI 0403E.b, Section B3.2.15
 * @see STM32F407 reference manual: RM0090, Section 10.4
 */
void HardFault_Handler(void);

/**
 * @brief Configurable Fault Status Register address.
 *
 * Composite register containing MemManage, BusFault, and UsageFault
 * status bits. Bit definitions:
 * - [25] DIVBYZERO: Integer division by zero (UsageFault)
 * - [24] UNALIGNED: Unaligned memory access (UsageFault)
 * - [15] BFARVALID: BFAR contains valid fault address (BusFault)
 * - [9]  PRECISERR: Precise data bus error (BusFault)
 * - [7]  MMARVALID: MMFAR contains valid fault address (MemManage)
 * - [1]  DACCVIOL:  Data access violation (MemManage)
 */
#define CFSR  (*((volatile uint32_t*)0xE000ED28U))
```

### A Documentation Checklist

Bringing the layers together, a practical checklist for documentation completeness in a production robotics module covers the following. Every public function and class in a header file has a Doxygen comment with at minimum a `@brief`, documented parameters, a documented return value, and explicit preconditions where they exist. Every non-obvious implementation choice in a function body has a comment explaining why, not what. Every module has a `README.md` covering its purpose, dependencies, and real-time constraints if applicable. Every significant architectural decision that is not obvious from the code has an ADR. The Doxygen build runs in CI and produces zero warnings.

None of these are onerous individually. Together they produce a codebase where a new engineer can become productive within days rather than weeks, where safety audits do not require reconstructing the rationale for design decisions from memory, and where the code and its documentation are genuinely synchronized rather than gradually diverging.
